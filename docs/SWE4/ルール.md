# SWE.4 ソフトウェアユニット検証 — ドキュメントルール

## 1. プロセス概要

### 目的

SDD に定義されたユニット設計が正しく実装されていることを**ユニット単位で検証**する。  
ユニットテストにより設計適合性・コードカバレッジを確認し、後続の統合テストへの品質基盤を提供する。

### 位置づけ（入出力）

```
入力:
  - ソフトウェア詳細設計書 (SDD)
  - ソースコード（ユニット）
  - コーディング規約

出力:
  - ユニットテスト仕様書
  - ユニットテスト結果報告書
  - カバレッジ計測結果

後続:
  - SWE.5（統合テストの前提として完了が必要）
```

## 4. ドキュメント記述ルール

### 4.1 ユニットテスト仕様書 (UTS)

#### 4.1.1 必須構成

| 章番号 | 章タイトル           | 必須記載内容                                       |
|--------|----------------------|----------------------------------------------------|
| 1      | 目的・スコープ       | 検証対象ユニット・テスト戦略の概要                  |
| 2      | 参照文書             | SDD・コーディング規約のリスト                      |
| 3      | 用語定義             | テスト固有の用語・略語                              |
| 4      | テストアプローチ     | 採用する検証手法（ホワイトボックス/ブラックボックス等） |
| 5      | テスト環境           | テストフレームワーク・ハードウェア・モック構成       |
| 6      | カバレッジ目標       | 命令・分岐・MC/DC 等の目標カバレッジ率             |
| 7      | テストケース         | 各テストケースの詳細仕様                           |
| 8      | テスト除外項目       | テストしない項目とその理由                          |
| 付録   | 変更履歴             | バージョン管理テーブル                             |

#### 4.1.2 テストケース記述ルール

各テストケースは以下のフィールドを持つ。

| フィールド   | ルール                                                        |
|--------------|---------------------------------------------------------------|
| ID           | `TC-UNT-<3桁連番>` 形式（例: TC-UNT-001）                   |
| テスト対象   | 対象ユニット ID (UNT-xxx) と対象関数/メソッド名              |
| 検証観点     | 正常系 / 異常系 / 境界値 / ゼロ値 / 最大値 のいずれかを明記 |
| 事前条件     | テスト実施前の環境・状態                                      |
| 入力         | 具体的な入力値（数値・型・単位を明記）                        |
| 期待結果     | 具体的な期待出力・状態変化（測定可能な条件）                  |
| 根拠         | 対応する SWR-xxx または UNT-xxx                              |
| 優先度       | High / Medium / Low                                          |

**テストケース設計の必須観点**:

| 観点           | 要件                                                    |
|----------------|---------------------------------------------------------|
| 正常系         | 典型的な入力での正常動作を検証する                      |
| 異常系         | 無効入力・エラー条件でのエラー処理を検証する            |
| 境界値         | 有効範囲の上限・下限・境界±1 を検証する                |
| 初期化         | 変数・状態の初期値を検証する                            |

#### 4.1.3 カバレッジ目標

| カバレッジ種別   | 目標値 | 注記                            |
|------------------|--------|---------------------------------|
| 命令カバレッジ   | 100%   | 到達不可コードは理由を記載      |
| 分岐カバレッジ   | 100%   | 到達不可分岐は理由を記載        |
| MC/DC            | 適用時 | ASIL B以上に適用                |

### 4.2 ユニットテスト結果報告書 (UTR)

#### 4.2.1 必須構成

| 章番号 | 章タイトル           | 必須記載内容                                  |
|--------|----------------------|-----------------------------------------------|
| 1      | テスト概要           | テスト実施日・環境・対象バージョン            |
| 2      | テスト実施サマリー   | 実施件数・合格/不合格/未実施の集計            |
| 3      | テストケース結果     | 各テストケースの合否・実測値                  |
| 4      | カバレッジ結果       | 実測カバレッジ率・未カバー箇所のリスト        |
| 5      | 不具合一覧           | 発見した不具合とその詳細                      |
| 6      | 評価・所見           | 目標達成状況・リスク評価・次工程への推奨事項  |
| 付録   | 変更履歴             | バージョン管理テーブル                        |

#### 4.2.2 テストケース結果の記述形式

| TC-ID       | テスト対象    | 判定   | 実測値 | 不具合ID | 備考 |
|-------------|--------------|--------|-------|---------|------|
| TC-UNT-001  | UNT-001      | PASS   | 0     | -       |      |
| TC-UNT-002  | UNT-001      | FAIL   | -1    | BUG-001 |      |

判定: **PASS** / **FAIL** / **BLOCKED** / **N/A**

#### 4.2.3 不具合記述ルール

| フィールド   | ルール                                              |
|--------------|-----------------------------------------------------|
| ID           | `BUG-<3桁連番>` 形式                               |
| 発生 TC      | 発見したテストケース ID                             |
| 重要度       | Critical / Major / Minor                           |
| 症状         | 期待値・実測値・差異を具体的に記述                  |
| 推定原因     | 暫定的な原因（特定できない場合は「調査中」）        |
| ステータス   | Open / In Progress / Fixed / Closed                |

## 5. トレーサビリティ要件

### 上位トレーサビリティ (UNT → TC-UNT)

- すべての UNT に対して少なくとも1つのテストケースが存在すること
- すべての SWR に対して少なくとも1つのテストケースが間接的に存在すること

### トレーサビリティマトリクス形式

| UNT-ID  | ユニット名 | TC-UNT-ID       | テストケースタイトル | 判定 |
|---------|-----------|-----------------|--------------------|----|
| UNT-001 | xxx       | TC-UNT-001      | 正常入力の検証      | PASS |

## 6. レビュー・承認基準

### レビューチェックリスト（テスト仕様書）

- [ ] すべての UNT に対してテストケースが設計されている
- [ ] 正常系・異常系・境界値の観点が網羅されている
- [ ] テストケースに期待結果が定量的に記述されている
- [ ] カバレッジ目標が明記されている
- [ ] テスト環境・構成が明記されている

### レビューチェックリスト（テスト結果報告書）

- [ ] すべてのテストケースに判定結果が記入されている
- [ ] カバレッジ目標が達成されている（未達の場合は根拠・対策を記載）
- [ ] FAIL / BLOCKED はすべて不具合として記録されている
- [ ] 重大・Critical 不具合がすべてクローズしている

### 承認条件

1. 命令・分岐カバレッジが目標値を達成していること
2. Critical / Major 不具合がすべてクローズしていること
3. 重大レビュー指摘がゼロであること
