# SWE.5 ソフトウェア統合・統合テスト — ドキュメントルール

## 1. プロセス概要

### 目的

SWE.4 で検証済みの各ユニットを**SAD に従い順次統合**し、  
統合したソフトウェアがアーキテクチャ設計・インターフェース仕様どおりに動作することを検証する。

### 位置づけ（入出力）

```
入力:
  - ソフトウェアアーキテクチャ設計書 (SAD)
  - インターフェース設計書
  - ユニットテスト結果報告書（全ユニット合格済み）

出力:
  - ソフトウェア統合計画書
  - 統合テスト仕様書
  - 統合テスト結果報告書

後続:
  - SWE.6（統合済みソフトウェアを適格性確認テストへ）
```

## 2. ベースプラクティス

| BP  | 内容                                           | 対応する成果物・記述箇所              |
|-----|------------------------------------------------|---------------------------------------|
| BP1 | 統合戦略の開発                                 | ソフトウェア統合計画書                |
| BP2 | ソフトウェア統合テスト仕様の開発               | 統合テスト仕様書                      |
| BP3 | ソフトウェアコンポーネントの統合                | 統合ビルド記録・統合計画書            |
| BP4 | ソフトウェア統合テストの実施                   | 統合テスト結果報告書                  |
| BP5 | 双方向トレーサビリティの確立                   | トレーサビリティマトリクス            |
| BP6 | 一貫性の確保                                   | 設計との対応確認記録                  |
| BP7 | 結果の要約・報告                               | 統合テスト結果報告書 サマリー章       |

## 3. 必須ドキュメント一覧

| ドキュメント名           | 略称  | 目的                                             |

## 4. ドキュメント記述ルール

### 4.1 ソフトウェア統合計画書 (SIP)

#### 4.1.1 必須構成

| 章番号 | 章タイトル           | 必須記載内容                                         |
|--------|----------------------|------------------------------------------------------|
| 1      | 目的・スコープ       | 統合の目的・対象コンポーネント範囲                    |
| 2      | 参照文書             | SAD・UTS 結果のリスト                               |
| 3      | 統合戦略             | ボトムアップ/トップダウン/ビッグバン等の方針・根拠    |
| 4      | 統合順序             | コンポーネント統合の順序と各ステップの条件            |
| 5      | ビルド構成           | 各統合ビルドのコンポーネント構成・バージョン         |
| 6      | 前提条件・依存関係   | 統合開始前に満たすべき条件（ユニットテスト合格等）   |
| 7      | リスクと対策         | 統合上のリスク・スタブ/ドライバの使用計画            |
| 付録   | 変更履歴             | バージョン管理テーブル                               |

#### 4.1.2 統合順序の記述形式

```markdown
| ステップ | 統合対象               | 前提（完了済み）    | スタブ/ドライバ | 完了基準 |
|---------|----------------------|-------------------|--------------|---------|
| INT-01  | ARC-001 + ARC-002    | UNT-001〜003 合格 | なし          | インターフェーステスト合格 |
| INT-02  | INT-01 + ARC-003     | INT-01 完了       | ARC-004 スタブ | インターフェーステスト合格 |
```

### 4.2 統合テスト仕様書 (ITS)

#### 4.2.1 必須構成

| 章番号 | 章タイトル           | 必須記載内容                                         |
|--------|----------------------|------------------------------------------------------|
| 1      | 目的・スコープ       | 検証対象インターフェース・機能連携の範囲              |
| 2      | 参照文書             | SAD・SIP・インターフェース設計書のリスト             |
| 3      | テストアプローチ     | 検証戦略・テストツール・環境構成                      |
| 4      | テスト環境           | HW 構成・スタブ/ドライバ・ネットワーク構成           |
| 5      | テストケース         | 各統合テストケースの詳細仕様                         |
| 6      | テスト除外項目       | テストしない項目と理由                               |
| 付録   | 変更履歴             | バージョン管理テーブル                               |

#### 4.2.2 統合テストケース記述ルール

| フィールド   | ルール                                                        |
|--------------|---------------------------------------------------------------|
| ID           | `TC-INT-<3桁連番>` 形式（例: TC-INT-001）                   |
| テスト対象   | 統合対象ステップ（INT-xx）と検証するインターフェース (IFC-xxx) |
| 検証観点     | 正常系 / 異常系 / 境界値 / タイミング / エラー伝搬 のいずれか |
| 事前条件     | テスト前のシステム状態・入力データ                            |
| 手順         | ステップ形式で操作手順を記述                                  |
| 期待結果     | 定量的な期待出力・状態                                        |
| 根拠         | 対応する SWR-xxx または IFC-xxx                              |
| 優先度       | High / Medium / Low                                          |

**統合テストの必須観点**:

| 観点             | 要件                                              |
|------------------|---------------------------------------------------|
| インターフェース | すべての IFC に対して正常系テストを実施する       |
| エラー伝搬       | コンポーネント間のエラー通知が正しく伝達されるか  |
| タイミング       | 非同期通信・タイムアウト処理を検証する            |
| データ整合性     | コンポーネント間でデータが正確に受け渡されるか    |

### 4.3 統合テスト結果報告書 (ITR)

#### 4.3.1 必須構成

| 章番号 | 章タイトル           | 必須記載内容                                   |
|--------|----------------------|------------------------------------------------|
| 1      | テスト概要           | テスト実施日・環境・対象バージョン              |
| 2      | テスト実施サマリー   | 実施件数・合格/不合格/未実施の集計             |
| 3      | テストケース結果     | 各テストケースの合否・実測値                   |
| 4      | 不具合一覧           | 発見した不具合の詳細                           |
| 5      | 回帰テスト結果       | 不具合修正後の再テスト結果                     |
| 6      | 評価・所見           | 目標達成状況・次工程への推奨事項               |
| 付録   | 変更履歴             | バージョン管理テーブル                         |

## 5. トレーサビリティ要件

### 上位トレーサビリティ (SAD IFC → TC-INT)

- すべての IFC に対して少なくとも1つの統合テストケースが存在すること

### トレーサビリティマトリクス形式

| IFC-ID  | インターフェース説明 | TC-INT-ID    | テストケースタイトル | 判定 |
|---------|-------------------|--------------|--------------------|------|
| IFC-001 | ARC-001↔ARC-002  | TC-INT-001   | 正常データ受け渡し  | PASS |

---

## 6. レビュー・承認基準

### レビューチェックリスト（統合テスト仕様書）

- [ ] すべての IFC に対してテストケースが設計されている
- [ ] エラー伝搬・タイミング観点のテストが含まれている
- [ ] スタブ/ドライバの仕様が明記されている
- [ ] 統合順序と前提条件が記載されている

### レビューチェックリスト（統合テスト結果報告書）

- [ ] すべてのテストケースに判定結果が記入されている
- [ ] FAIL 件数がゼロ、または不具合がすべてクローズしている
- [ ] 回帰テストが実施されている

### 承認条件

1. すべての統合テストケースが PASS であること
2. Critical / Major 不具合がすべてクローズしていること
3. 工程責任者の署名（または承認記録）があること
