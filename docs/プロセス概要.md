## 1. プロセス全体像

ASPICE SWE プロセスは V 字モデルに対応した 6 工程で構成される。  
左辺（SWE.1〜SWE.3）で設計を作り込み、右辺（SWE.4〜SWE.6）でそれぞれに対応する検証を行う。

```
SWE.1 要件分析 ────────────────────────── SWE.6 適格性確認テスト
  │  SRS → 要件を確立                          ↑ SWR 全件の充足確認
  ↓                                            │
SWE.2 アーキテクチャ設計 ─────────── SWE.5 統合テスト
  │  SAD → コンポーネントに分割                ↑ コンポーネント間 IF 検証
  ↓                                            │
SWE.3 詳細設計・実装 ────────── SWE.4 ユニット検証
       SDD + ソースコード              ↑ ユニット単位の設計適合確認
```

## 2. 各工程サマリー

### SWE.1 ソフトウェア要件分析

> 詳細ルール → [SWE1/ルール.md](./SWE1/ルール.md)

**役割**  
システム要件をソフトウェアの観点で詳細化し、開発チームが実装に使える要件として確立する。「何を作るか」を定義する工程。

**入力**

| ドキュメント               | 供給元       |
|---------------------------|--------------|
| システム要件仕様書 (SYS)  | システム工程 |
| システムアーキテクチャ設計書 | システム工程 |
| インターフェース要件定義   | システム工程 |

**成果物**

| 略称 | ドキュメント名             | 主な記載内容                                        |
|------|----------------------------|-----------------------------------------------------|
| SRS  | ソフトウェア要件仕様書     | 機能要件 (SWR-xxx)・非機能要件・インターフェース要件 |
| IRS  | インターフェース要件定義書 | 外部システム・HW との接続仕様                       |
| TM   | トレーサビリティマトリクス | SYS 要件 → SWR の対応表                            |

**要素 ID**

| 要素             | ID 形式      | 例      |
|------------------|--------------|---------|
| ソフトウェア要件 | `SWR-<3桁>` | SWR-001 |

### SWE.2 ソフトウェアアーキテクチャ設計

> 詳細ルール → [SWE2/ルール.md](./SWE2/ルール.md)

**役割**  
SRS の要件を実現するためのソフトウェア構造を設計する。コンポーネント分割・インターフェース・動的ふるまいを定義し、SWE.3 の実装基盤を提供する。「どう分割して作るか」を定義する工程。

**入力**

| ドキュメント                     | 供給元 |
|----------------------------------|--------|
| ソフトウェア要件仕様書 (SRS)    | SWE.1  |
| インターフェース要件定義書       | SWE.1  |

**成果物**

| 略称 | ドキュメント名                   | 主な記載内容                                            |
|------|----------------------------------|---------------------------------------------------------|
| SAD  | ソフトウェアアーキテクチャ設計書 | コンポーネント構成 (ARC-xxx)・IF 設計・動的ふるまい・ADR |
| IFD  | インターフェース設計書           | コンポーネント間 API・データ形式・通信方式 (IFC-xxx)    |
| TM   | トレーサビリティマトリクス       | SWR → ARC の対応表                                     |

**要素 ID**

| 要素                         | ID 形式       | 例      |
|------------------------------|---------------|---------|
| アーキテクチャコンポーネント | `ARC-<3桁>`  | ARC-001 |
| インターフェース             | `IFC-<3桁>`  | IFC-001 |

### SWE.3 ソフトウェア詳細設計・ユニット構築

> 詳細ルール → [SWE3/ルール.md](./SWE3/ルール.md)

**役割**  
SAD のコンポーネントを実装可能な最小単位（ユニット）に詳細化し、ソースコードを実装する。設計と実装を同時に完成させる工程。

**入力**

| ドキュメント                         | 供給元 |
|--------------------------------------|--------|
| ソフトウェアアーキテクチャ設計書 (SAD) | SWE.2 |
| インターフェース設計書               | SWE.2  |

**成果物**

| 略称 | ドキュメント名             | 主な記載内容                                                         |
|------|----------------------------|----------------------------------------------------------------------|
| SDD  | ソフトウェア詳細設計書     | ユニット設計 (UNT-xxx)・構造図・動的ふるまい図・API 仕様・エラー処理 |
| COD  | コーディング規約           | 命名規則・禁止事項・静的解析設定                                     |
| —    | ソースコード               | 各ユニットの実装（UNT-ID をコメントに記載）                          |
| TM   | トレーサビリティマトリクス | ARC → UNT の対応表                                                  |

**SDD に含める図（PlantUML 記法）**

| 図の種類             | 必須/推奨 | 適用条件                          |
|----------------------|-----------|-----------------------------------|
| モジュール構造図     | **必須**  | 常に                              |
| クラス図             | **必須**  | OOP 言語を使う場合               |
| データ構造図         | **必須**  | C 言語で struct/typedef を使う場合|
| シーケンス図（正常系）| **必須** | ユニット間メッセージが発生する機能 |
| シーケンス図（異常系）| **必須** | エラー・タイムアウト等の異常フロー |
| 状態遷移図           | **必須**  | 状態を持つユニットが存在する場合  |
| フローチャート       | **必須**  | 循環的複雑度 > 5 の関数           |
| ER 図                | 推奨      | DB・永続化データ構造がある場合    |

**図の粒度ポリシー**（SDD 作成前に決定・宣言する）

| レベル | 名称       | クラス図                               | シーケンス図                               |
|--------|------------|----------------------------------------|--------------------------------------------|
| L1     | 概念レベル | クラス名・関係のみ                     | 参加者名・メッセージ名のみ                 |
| **L2** | **仕様レベル** | **L1 ＋ public メンバー全件（型付き）** | **L1 ＋ 引数・戻り値・alt/loop ブロック** |
| L3     | 実装レベル | L2 ＋ private/protected 全件           | L2 ＋ 型付きシグネチャ・private 呼び出し  |

> ASPICE SWE.3 の最低要件は **L2**。詳細は [SWE3/ルール.md § 4.1.2](./SWE3/ルール.md) を参照。

**要素 ID**

| 要素         | ID 形式      | 例      |
|--------------|--------------|---------|
| 設計ユニット | `UNT-<3桁>` | UNT-001 |

### SWE.4 ソフトウェアユニット検証

> 詳細ルール → [SWE4/ルール.md](./SWE4/ルール.md)

**役割**  
SDD に定義した設計が正しく実装されていることをユニット単位で検証する。コードカバレッジを計測し、後続の統合テストへの品質基盤を確立する工程。

**入力**

| ドキュメント                 | 供給元 |
|------------------------------|--------|
| ソフトウェア詳細設計書 (SDD) | SWE.3  |
| ソースコード                 | SWE.3  |
| コーディング規約             | SWE.3  |

**成果物**

| 略称 | ドキュメント名           | 主な記載内容                                          |
|------|--------------------------|-------------------------------------------------------|
| UTS  | ユニットテスト仕様書     | テストケース (TC-UNT-xxx)・カバレッジ目標・テスト環境 |
| UTR  | ユニットテスト結果報告書 | 合否結果・実測カバレッジ・不具合 (BUG-xxx) 一覧      |

**要素 ID**

| 要素               | ID 形式         | 例         |
|--------------------|-----------------|------------|
| ユニットテストケース | `TC-UNT-<3桁>` | TC-UNT-001 |
| 不具合             | `BUG-<3桁>`    | BUG-001    |

**カバレッジ目標**

| 種別           | 目標値           |
|----------------|------------------|
| 命令カバレッジ | 100%             |
| 分岐カバレッジ | 100%             |
| MC/DC          | ASIL B 以上に適用 |

## 4. トレーサビリティチェーン

要件から検証まで、すべての要素が双方向に追跡できることを必須とする。

```
SYS 要件 (SYS-xxx)
    │ 割り当て
    ▼
SW 要件 (SWR-xxx)            ─── SWE.1 ───── 適格性確認 TC (TC-QAL-xxx)
    │ 設計                                          │ SWE.6
    ▼                                               │
アーキテクチャ (ARC-xxx)     ─── SWE.2 ─── 統合 TC (TC-INT-xxx)
    │ 詳細化                                        │ SWE.5
    ▼                                               │
設計ユニット (UNT-xxx)       ─── SWE.3 ─── ユニット TC (TC-UNT-xxx)
                                                    │ SWE.4
```

各工程間の接続要件:

| 接続         | 要件                                                      |
|--------------|-----------------------------------------------------------|
| SYS → SWR    | 全 SWR に対応 SYS-ID を記載（派生の場合は根拠を記録）   |
| SWR → ARC    | 全 SWR を少なくとも 1 つの ARC に割り当てる             |
| ARC → UNT    | 全 ARC を少なくとも 1 つの UNT に分解する               |
| UNT → TC-UNT | 全 UNT に少なくとも 1 つのテストケースを設計する         |
| IFC → TC-INT | 全 IFC に少なくとも 1 つの統合テストケースを設計する     |
| SWR → TC-QAL | 全 Must SWR に少なくとも 1 つの TC-QAL を設計する       |

## 5. 工程間ゲート条件

次工程に進む前に必ず満たすべき条件。

| 工程完了    | 次工程      | ゲート条件                                                   |
|-------------|-------------|--------------------------------------------------------------|
| SWE.1 完了  | SWE.2 開始  | SRS が承認済み / 重大レビュー指摘ゼロ                        |
| SWE.2 完了  | SWE.3 開始  | SAD が承認済み / 全 SWR が ARC に割り当て済み               |
| SWE.3 完了  | SWE.4 開始  | SDD 承認済み（粒度ポリシー宣言含む）/ コーディング規約承認済み / 静的解析通過 |
| SWE.4 完了  | SWE.5 開始  | 全ユニット TC が PASS / カバレッジ目標達成                   |
| SWE.5 完了  | SWE.6 開始  | 全統合 TC が PASS / Critical・Major 不具合クローズ           |
| SWE.6 完了  | リリース    | 全 Must SWR が TC-QAL で PASS / リリース判定「可」          |
