# SWE.1 ソフトウェア要件分析 — ドキュメントルール

## 1. プロセス概要

### 目的

システム要件仕様書・システムアーキテクチャ設計書、およびインターフェース要件から、  
**ソフトウェアに割り当てられた要件を分析・詳細化**し、ソフトウェア要件仕様書 (SRS) として確立する。

### 位置づけ（入出力）

```
入力:
  - システム要件仕様書 (SYS 要件)
  - システムアーキテクチャ設計書
  - インターフェース要件定義

出力:
  - ソフトウェア要件仕様書 (SRS)
  - インターフェース要件定義書（SW 観点）
  - トレーサビリティマトリクス（SYS → SWR）

後続:
  - SWE.2（SRS を入力としてアーキテクチャ設計へ）
```

## 2. ベースプラクティス

| BP  | 内容                                           | 対応する成果物・記述箇所          |
|-----|------------------------------------------------|-----------------------------------|
| BP1 | ソフトウェア要件の仕様化                       | SRS 機能要件・非機能要件章        |
| BP2 | 要件の実現可能性評価                           | SRS 実現可能性評価欄              |
| BP3 | ハードウェア・インターフェース要件の分析       | SRS インターフェース要件章        |
| BP4 | 要件の優先度付け                               | SRS 各要件の優先度フィールド      |
| BP5 | ピア要素とのインターフェース合意               | インターフェース要件定義書        |
| BP6 | ステークホルダーとの要件合意                   | SRS 承認欄・議事録                |
| BP7 | 双方向トレーサビリティの確立                   | トレーサビリティマトリクス        |
| BP8 | 非一貫性の識別                                 | SRS 課題・懸念事項リスト          |
| BP9 | 合意のサポート                                 | レビュー記録・承認記録            |

## 4. ドキュメント記述ルール

### 4.1 ソフトウェア要件仕様書 (SRS)

#### 4.1.1 必須構成

| 章番号 | 章タイトル           | 必須記載内容                                     |
|--------|----------------------|--------------------------------------------------|
| 1      | 目的・スコープ       | 本 SRS の適用範囲、対象システム、対象外事項      |
| 2      | 参照文書             | 依拠する上位仕様書・標準規格のリスト             |
| 3      | 用語定義             | 本書で使用する略語・専門用語の定義               |
| 4      | システム概要         | ソフトウェアが動作するシステム環境の記述         |
| 5      | 機能要件             | 各機能要件を ID・説明・優先度・受入基準で記述   |
| 6      | 非機能要件           | 性能・安全・セキュリティ・信頼性等の要件         |
| 7      | インターフェース要件 | HW/外部システム/ユーザとのインターフェース仕様  |
| 8      | 制約条件             | 設計・実装上の制約（規格・ツール・言語等）       |
| 9      | トレーサビリティ     | SYS 要件 → SWR のマトリクス                     |
| 10     | 課題・懸念事項       | 未解決事項・想定リスク・決定待ち事項             |
| 付録   | 変更履歴             | バージョン管理テーブル                           |

#### 4.1.2 機能要件の記述ルール

各機能要件は以下のフィールドを持つ。

| フィールド   | ルール                                                   |
|--------------|----------------------------------------------------------|
| ID           | `SWR-<3桁連番>` 形式（例: SWR-001）                    |
| タイトル     | 主語＋動詞で30字以内                                     |
| 説明         | 「システムは〜しなければならない (shall)」形式で記述    |
| 優先度       | Must / Should / Nice-to-have の3段階                   |
| 根拠 (上位ID)| 対応するシステム要件 ID (SYS-xxx) を必ず記載            |
| 受入基準     | 検証可能な合否判定条件を定量的に記述                     |
| ステータス   | Draft / Agreed / Deleted のいずれか                     |

**禁止事項**:
- 「速く動作する」「使いやすい」など測定不能な表現
- 1つの要件 ID に複数の要件を混在させること（1 ID = 1 要件）
- 実装方法の指定（HOW ではなく WHAT を記述する）

#### 4.1.3 非機能要件の記述ルール

| 種別       | 必須記載項目                               |
|------------|--------------------------------------------|
| 性能       | 応答時間・スループット・数値目標値         |
| 安全       | ASIL レベル・フェールセーフ要件            |
| セキュリティ | 脅威モデル・保護対象・対策レベル          |
| 信頼性     | MTBF・可用率・フォールトトレランス        |
| 保守性     | 診断機能・ログ要件・アップデート機能       |

## 5. トレーサビリティ要件

### 上位トレーサビリティ (SYS → SWR)

- すべての SWR は対応する SYS 要件 ID を必ず持つこと
- SYS 要件から導出された場合は `SYS-xxx から派生` と明記する
- 対応する SYS 要件が存在しない SWR は、追加根拠（安全要件・規格適合等）を記載する

### 下位トレーサビリティ (SWR → SWE.2)

- SWE.2 完了後、各 SWR に対応するアーキテクチャ要素 (ARC-xxx) を SRS に追記する

### トレーサビリティマトリクス形式

| SYS-ID  | SYS 要件タイトル | SWR-ID  | SWR タイトル     | 備考         |
|---------|-----------------|---------|-----------------|--------------|
| SYS-001 | ○○機能          | SWR-001 | ○○処理機能      | 完全割り当て |
| SYS-001 | ○○機能          | SWR-002 | ○○エラー処理    | 部分割り当て |

## 6. レビュー・承認基準

### レビューチェックリスト

- [ ] すべての SWR に一意の ID が付与されている
- [ ] すべての SWR に対応する SYS-ID が記載されている
- [ ] SWR の説明が "shall" 形式（測定可能）で記述されている
- [ ] 受入基準が定量的に記述されている
- [ ] インターフェース要件が網羅されている
- [ ] 非機能要件に数値目標が含まれている
- [ ] 課題・懸念事項リストに未解決事項が記載されている
- [ ] すべての SYS 要件に対して少なくとも1つの SWR が存在する
- [ ] 変更履歴テーブルが最新状態になっている

### 承認条件

1. 上記チェックリストの全項目が ✓ であること
2. 重大レビュー指摘がゼロであること
3. 工程責任者の署名（または承認記録）があること

## 7. 成果物管理ルール

| 項目           | ルール                                              |
|----------------|-----------------------------------------------------|
| ファイル命名   | `SRS_<プロジェクトID>_<連番>_v<X.Y>.md`            |
| 格納場所       | `docs/SWE1/` 配下                                  |
| ドラフト管理   | ファイル名末尾に `-draft<N>` を付与                |
| 承認後の変更   | バージョンを上げ、旧版を `archive/` に移動する     |
| 削除禁止       | 承認済み版は削除せず、ステータスを "Superseded" に |
