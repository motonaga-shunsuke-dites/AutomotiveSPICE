# 全工程共通ルール

## 1. 適用範囲

本ドキュメントに定めるルールは、SWE.1〜SWE.6 のすべての工程に適用する。  
各工程固有のルールは各工程の `ルール.md` に定める。両者が矛盾する場合は、工程固有ルールを優先する。

---

## 2. ドキュメント管理

### 2.1 ファイル命名規則

```
<ドキュメント種別略称>_<プロジェクトID>_<連番>_<バージョン>.<拡張子>

例: SRS_PRJ001_001_v1.0.md
    SDD_PRJ001_003_v2.1.md
```

| フィールド         | 規則                                          |
|--------------------|-----------------------------------------------|
| ドキュメント種別略称 | SRS / SAD / SDD / UTS / UTR / ITS / ITR / QTS / QTR |
| プロジェクトID      | 英数字3〜8文字                               |
| 連番               | 3桁ゼロ埋め (001〜999)                       |
| バージョン         | `v<メジャー>.<マイナー>` 形式               |
| 拡張子             | `.md`（Markdown）                            |

### 2.2 バージョン管理規則

| 変更種別         | バージョン変更         | 例             |
|------------------|------------------------|----------------|
| 初版リリース     | v1.0                   | v1.0           |
| 軽微な修正・追記 | マイナーをインクリメント | v1.0 → v1.1   |
| 構成変更・大幅改訂 | メジャーをインクリメント | v1.1 → v2.0  |
| 作業中ドラフト   | ドラフト番号を付与     | v1.0-draft1    |

### 2.3 変更履歴の記録

すべてのドキュメントは末尾に変更履歴テーブルを持つ。

```markdown
## 変更履歴

| バージョン | 日付       | 変更概要     | 作成/変更者 | レビュー者 |
|-----------|------------|--------------|------------|-----------|
| 1.0       | YYYY-MM-DD | 初版作成     |            |           |
```

## 4. ID 体系

### 4.1 各要素の ID 形式

| 対象             | ID フォーマット         | 例             |
|------------------|------------------------|----------------|
| システム要件     | `SYS-<3桁連番>`        | SYS-001        |
| ソフトウェア要件 | `SWR-<3桁連番>`        | SWR-001        |
| アーキテクチャコンポーネント | `ARC-<3桁連番>` | ARC-001  |
| 詳細設計ユニット | `UNT-<3桁連番>`        | UNT-001        |
| テストケース     | `TC-<工程略称>-<3桁連番>` | TC-UNT-001  |
| テスト結果       | `TR-<工程略称>-<3桁連番>` | TR-UNT-001  |

### 4.2 ID 管理規則

- ID は一度割り当てたら変更・再利用しない
- 削除された要素の ID は欠番とし、削除理由を記録する
- 各工程の担当者が自工程の ID 採番を管理する

## 5. トレーサビリティ

### 5.1 トレーサビリティチェーン

```
SYS要件 (SYS-xxx)
    ↓ 割り当て
SW要件 (SWR-xxx)          ← SWE.1
    ↓ 設計
アーキテクチャ要素 (ARC-xxx) ← SWE.2
    ↓ 詳細化
設計ユニット (UNT-xxx)     ← SWE.3
    ↓ 検証
ユニットTC (TC-UNT-xxx)    ← SWE.4
統合TC (TC-INT-xxx)        ← SWE.5
適格性確認TC (TC-QAL-xxx)  ← SWE.6
```

### 5.2 トレーサビリティマトリクス

各工程のドキュメントには、上位要素と下位要素の対応を示すトレーサビリティマトリクスを含める。

| 上位ID  | 上位説明     | 下位ID  | 下位説明     | 備考 |
|---------|-------------|---------|-------------|------|
| SWR-001 | ○○機能の要件 | ARC-003 | ○○モジュール | -    |

- **双方向性**: 上位→下位（割り当て）と下位→上位（根拠）の両方を確保する
- **完全性**: すべての上位要素に対して少なくとも1つの下位要素が存在することを確認する
- **一貫性**: 上位要素の変更時は関連する下位要素への影響を評価し記録する

## 6. レビュー規則

### 6.1 レビュー種別

| 種別           | タイミング             | 参加者                     |
|----------------|------------------------|----------------------------|
| 作成者自己レビュー | ドキュメント完成後    | 作成者本人                 |
| ピアレビュー   | 自己レビュー後         | 同工程の別担当者 (1名以上)  |
| 承認レビュー   | ピアレビュー後         | 工程責任者                 |

### 6.2 レビュー記録

レビュー指摘事項は以下の形式で記録する。

```markdown
| 指摘ID | 重要度 | 箇所       | 指摘内容     | 対応方針     | 完了日     |
|--------|--------|------------|-------------|-------------|------------|
| R-001  | 重大   | 3.2節      | ○○が不明確  | ○○に修正    | YYYY-MM-DD |
```

重要度: **重大**（承認前に必ず修正）/ **中**（次版までに修正）/ **軽微**（任意対応）

### 6.3 承認条件

- 重大指摘がゼロであること
- トレーサビリティマトリクスに欠落がないこと
- 必須記載項目がすべて埋まっていること
