# SWE.6 ソフトウェア適格性確認テスト — ドキュメントルール

## 1. プロセス概要

### 目的

統合済みソフトウェアが**SRS に定義されたすべてのソフトウェア要件を満たすこと**を確認する。  
顧客・ステークホルダーの合意のもとで適格性確認を実施し、ソフトウェアのリリース判定基盤を提供する。

### 位置づけ（入出力）

```
入力:
  - ソフトウェア要件仕様書 (SRS)
  - 統合テスト結果報告書（合格済み）
  - 統合済みソフトウェア

出力:
  - 適格性確認テスト仕様書
  - 適格性確認テスト結果報告書

後続:
  - システムテスト（SW → システムへの引き渡し）
  - リリース判定
```

## 4. ドキュメント記述ルール

### 4.1 適格性確認テスト仕様書 (QTS)

#### 4.1.1 必須構成

| 章番号 | 章タイトル               | 必須記載内容                                           |
|--------|--------------------------|--------------------------------------------------------|
| 1      | 目的・スコープ           | 検証対象 SWR の範囲・テスト戦略の概要                  |
| 2      | 参照文書                 | SRS・統合テスト結果・ITS のリスト                     |
| 3      | 用語定義                 | テスト固有の用語・略語                                 |
| 4      | テストアプローチ         | ブラックボックス/グレーボックス・環境・ツール          |
| 5      | テスト環境               | HW 構成・ソフトウェアバージョン・ネットワーク構成      |
| 6      | テストケース             | 各テストケースの詳細仕様                               |
| 7      | テスト要件カバレッジ     | SWR ごとのテストケース割り当て一覧                     |
| 8      | テスト除外項目           | テストしない SWR とその理由                            |
| 付録   | 変更履歴                 | バージョン管理テーブル                                 |

#### 4.1.2 適格性確認テストケース記述ルール

| フィールド   | ルール                                                        |
|--------------|---------------------------------------------------------------|
| ID           | `TC-QAL-<3桁連番>` 形式（例: TC-QAL-001）                   |
| テスト対象   | 検証する SWR-xxx（1テストケースに複数 SWR 可）               |
| 検証観点     | 機能確認 / 性能確認 / 境界値 / 異常系 / セキュリティ 等      |
| 事前条件     | テスト前のシステム状態・初期データ                            |
| 手順         | ステップ形式で操作手順を記述（番号付き）                      |
| 期待結果     | 合否判定が明確な期待出力・状態（定量値を含む）                |
| 根拠         | 対応する SWR-xxx                                             |
| 優先度       | High / Medium / Low                                          |
| テスト種別   | 手動 / 自動 / 半自動                                         |

**適格性確認テストの設計方針**:

| 観点           | 要件                                                     |
|----------------|----------------------------------------------------------|
| SWR 完全性     | すべての SWR（Must）に対して少なくとも1つの TC を設計する |
| 受入基準       | SRS に記載された受入基準をそのままテスト合否条件に使う   |
| 実環境         | 可能な限り実ターゲット環境でテストする                   |
| 独立性         | SWE.4/SWE.5 テスト担当者とは異なる担当者が実施することを推奨 |
| 回帰           | SWR を変更した場合、関連するすべての TC を再実施する     |

#### 4.1.3 非機能要件テストの記述追加要件

| 種別       | 追加記載内容                                              |
|------------|----------------------------------------------------------|
| 性能       | 計測ツール・計測点・計測回数・統計処理方法（平均・最悪値）|
| 安全       | フォルトインジェクション方法・評価基準                    |
| セキュリティ | 脅威シナリオ・攻撃ベクター・評価基準                    |

### 4.2 適格性確認テスト結果報告書 (QTR)

#### 4.2.1 必須構成

| 章番号 | 章タイトル               | 必須記載内容                                       |
|--------|--------------------------|----------------------------------------------------|
| 1      | テスト概要               | テスト実施日・環境・対象バージョン                  |
| 2      | テスト実施サマリー       | 実施件数・合格/不合格/未実施の集計                 |
| 3      | SWR カバレッジサマリー   | SWR ごとの検証状況・カバレッジ率                   |
| 4      | テストケース結果         | 各テストケースの合否・実測値                        |
| 5      | 不具合一覧               | 発見した不具合の詳細・クローズ状況                 |
| 6      | 回帰テスト結果           | 不具合修正後の再テスト結果                         |
| 7      | リリース判定             | SW リリースの可否判定・残課題・条件付き承認の場合の条件 |
| 付録   | 変更履歴                 | バージョン管理テーブル                             |

#### 4.2.2 SWR カバレッジサマリーの記述形式

| SWR-ID  | SWR タイトル | 優先度 | 割り当て TC-QAL | 判定   | 備考 |
|---------|-------------|--------|----------------|--------|------|
| SWR-001 | ○○処理機能  | Must   | TC-QAL-001     | PASS   |      |
| SWR-N01 | 応答時間     | Must   | TC-QAL-010     | PASS   | 最悪値 XX ms |

#### 4.2.3 リリース判定の記述形式

```markdown
## 7. リリース判定

### 判定結果: [リリース可 / 条件付きリリース可 / リリース不可]

| 評価項目                       | 状況    | 備考 |
|-------------------------------|---------|------|
| Must 要件のテスト合格率        | XX%     |      |
| Critical/Major 不具合クローズ  | 完了/残XX件 |   |
| SWR カバレッジ                 | XX%     |      |

### 残課題・条件
- <条件付きリリースの場合の承認条件>
```

## 5. トレーサビリティ要件

### SWR → TC-QAL の完全性確認

- SRS の「Must」SWR はすべて少なくとも1つの TC-QAL に対応すること
- 「Should」SWR も原則対応させ、除外する場合は理由を記録すること

### トレーサビリティマトリクス形式

| SWR-ID  | SWR タイトル | 優先度 | TC-QAL-ID    | テストケースタイトル   | 判定 |
|---------|-------------|--------|--------------|----------------------|------|
| SWR-001 | ○○処理機能  | Must   | TC-QAL-001   | ○○機能の動作確認      | PASS |

## 6. レビュー・承認基準

### レビューチェックリスト（テスト仕様書）

- [ ] すべての Must SWR に対してテストケースが設計されている
- [ ] テストケースの期待結果が定量的に記述されている
- [ ] 手順が第三者でも実施できる粒度で記述されている
- [ ] 性能・安全・セキュリティ要件の検証方法が記述されている
- [ ] SWR カバレッジ一覧が完全に埋まっている

### レビューチェックリスト（テスト結果報告書）

- [ ] すべての Must SWR が少なくとも1つの PASS テストで検証されている
- [ ] FAIL 件数がゼロ、または不具合がすべてクローズしている
- [ ] リリース判定が記載されている
- [ ] 回帰テストが実施されている

### 承認条件

1. Must SWR のテストカバレッジが 100% であること
2. Critical / Major 不具合がすべてクローズしていること
3. リリース判定が「リリース可」または承認済み条件付きリリース可であること
4. 工程責任者・品質管理者の署名（または承認記録）があること
