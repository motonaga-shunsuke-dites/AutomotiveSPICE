参照ルール: `docs/SWE2/ルール.md`

---

## ⚠️ 重要：SAD は人間が構造を決めてから AI に渡す

AIは「局所最適が得意、全体設計が苦手」。  
コンポーネント境界・責務・依存関係の**決定**は人間が行う。  
AIの役割は「人間が決めた構造を SAD 形式に清書・図示・ギャップ指摘すること」。

**やってはいけない依頼例**:
```
× 「このSRSをもとにアーキテクチャを設計してください」
```
→ AIが全体設計を試みる。コンポーネント分割の根拠・チームの実装能力・
  既存資産の考慮なしに「それらしい」設計を生成する。

**正しい依頼例**:
```
✓ 「以下の私が決めたコンポーネント一覧を SAD 形式に清書してください。
    ギャップや矛盾があれば指摘してください」
```

---

## 必要な Input 資料

| 資料 | 必須/推奨 | 内容 |
|------|----------|------|
| **人間が描いたコンポーネントスケッチ** | **必須** | コンポーネント名・責務・主要インタフェースのメモ |
| SRS（承認済み） | **必須** | 全SWR一覧（清書後に割り当て確認に使う） |
| システムアーキテクチャ概要 | 必須 | HW構成・ソフトウェアの配置環境 |
| インタフェース要件定義書 (IRS) | 推奨 | 外部接続仕様（SWE.1成果物） |
| 採用ミドルウェア仕様 | 推奨 | AUTOSAR RTE・OS API など |

## Step 1（人間作業）：コンポーネントスケッチを描く

AI に依頼する前に、以下の2点を紙・ホワイトボード・メモで決める。  
詳細は不要。**名前・責務・接続先** の3点だけでよい。

```
【コンポーネントスケッチ例】
1. AuthManager
   - 責務: トークン発行・検証・失効
   - 接続: UserRepository（利用）、外部認証サーバ（CAN経由）

2. UserRepository
   - 責務: ユーザ情報のDB読み書き
   - 接続: DB（EEPROM）

3. NetworkManager
   - 責務: CAN/Ethernet送受信・再送制御
   - 接続: CANドライバ（HW）

4. DiagManager
   - 責務: エラーコード管理・UDS診断応答
   - 接続: AuthManager, NetworkManager
```

**スケッチのポイント**:
- コンポーネント数は5〜10個程度が目安（多すぎたら階層化を検討）
- 循環依存がないか確認（A→B→A はNG）
- 1コンポーネント = 1チームまたは1開発者が担当できる粒度

## Step 2（AI 作業）：スケッチを SAD 形式に清書

### プロンプト：コンポーネント定義の清書・ギャップ指摘

```
以下の私が定義したコンポーネントスケッチを、
ASPICE SWE.2準拠の SAD 形式に清書してください。

また、以下の観点でギャップ・矛盾・懸念点があれば指摘してください：
- SRSのSWRが割り当てられていないコンポーネントがあるか
- どのコンポーネントにも割り当てられていないSWRがあるか
- コンポーネント間の依存方向に循環がないか
- 単一責任原則に明らかに違反しているコンポーネントがあるか

【プロジェクトコンテキスト】
- プロジェクトID    : [プロジェクトID]
- ターゲット環境    : [MCU / OS / ミドルウェア]
- アーキテクチャ方針: [例: 3層レイヤー / feature分割 / AUTOSAR SWC]

【ガードレール】
- 私が定義したコンポーネント以外を勝手に追加しないこと
- 追加が必要と判断した場合は「提案」として明示し、私の確認を求めること

【私が定義したコンポーネントスケッチ】
（Step1 で作成したメモをそのまま貼り付ける）

【SRS 要件一覧】
[SWR-ID] [要件タイトル]
（機能要件の全件）

【出力ルール】
■ 各コンポーネントの正式定義（SAD 章5形式）
  - ID: ARC-xxx（3桁連番）
  - コンポーネント名・責務・担当SWR・依存・インタフェース
■ PlantUML コンポーネント図（依存関係付き）
■ ギャップ指摘リスト（確認が必要な点を箇条書き）
```

---

### プロンプト：ギャップ修正後にインタフェース設計を清書

*人間が Step2 のギャップ指摘に対応してスケッチを修正した後に依頼する。*

```
以下の確定したコンポーネント定義をもとに、
コンポーネント間インタフェースを設計してください。

【確定したコンポーネント定義（ARC一覧）】
（Step2 で確定した ARC 定義を貼り付ける）

【外部インタフェース（IRS から）】
（IRS の外部接続仕様を貼り付ける）

【出力ルール】
各インタフェースを以下のフィールドで記述してください：
- インタフェースID: IFC-xxx
- 提供コンポーネント: ARC-xxx
- 消費コンポーネント: ARC-xxx
- 操作名: 動詞＋目的語（例: getTemperature()）
- 引数: 型・範囲・単位を明記
- 戻り値: 型・意味・エラーコード
- 呼び出しタイミング: 定期(周期ms) / イベント駆動(トリガー条件)
- エラー処理: タイムアウト・異常値時の動作

【ガードレール】
- コンポーネント定義に存在しないインタフェースを追加しないこと
- 提案がある場合は「[提案]」と明示して私の確認を求めること
```

### プロンプト：動的ふるまいの清書

*コンポーネントとインタフェースが確定した後に依頼する。*

```
以下の主要ユースケースについて、PlantUML シーケンス図を作成してください。

【確定したコンポーネント・インタフェース】
（ARC一覧 + IFC一覧を貼り付ける）

【描いてほしいユースケース（私が指定する）】
1. [ユースケース名: 例 システム起動シーケンス]
2. [ユースケース名: 例 ○○制御の正常フロー]
3. [ユースケース名: 例 通信エラー時のフェールセーフ動作]

【出力ルール】
各ユースケースについて：
1. 正常系シーケンス図（@startuml 〜 @enduml）
2. 異常系シーケンス図（エラー・タイムアウト発生時）
3. タイミング制約がある場合は note で ms 単位を明記

【ガードレール】
- 定義済みの ARC・IFC に存在しない呼び出しを描かないこと
- 描く必要があると判断した場合は「[提案]」と明示すること
```

### プロンプト：リソース消費目標の清書

```
以下の制約をもとに、各コンポーネントのリソース消費目標を設計してください。

【システム全体のリソース制約】
- CPU使用率上限  : [例: 70%（通常）/ 90%（最悪ケース）]
- RAM 上限        : [例: 64KB]
- ROM/Flash 上限  : [例: 256KB]
- スタックサイズ  : [例: タスクあたり最大 2KB]
- 通信帯域        : [例: CAN 500kbps、利用可能帯域 40%]

【確定したコンポーネント一覧（ARC）】
（ARC一覧を貼り付ける）

【出力ルール】
コンポーネントごとにリソース割り当て目標テーブルを作成してください。
最終行に合計と制約値との差分（マージン）を記載してください。
合計が制約を超える場合は「⚠️ 超過」と明示して確認を促してください。

【ガードレール】
- リソース割り当ての根拠・仮定を必ず記載すること
- 実測値ではなく目標値であることを明記すること
```

### プロンプト：ADR（設計決定記録）の清書

```
以下の設計決定を ADR 形式で記録してください。

【私が決定した内容】
（人間が記述した設計判断のメモを貼り付ける）
例:
- アーキテクチャ: 3層レイヤー型を採用（理由: AUTOSAR準拠のため）
- コンポーネント通信: 非同期メッセージキューを採用（理由: RTOS間のデカップリングのため）
- エラー処理: DiagManagerに集中管理（理由: UDS診断の一元化のため）

【出力ルール】
各決定を以下の ADR 形式で記述してください：
### ADR-001: [決定事項タイトル]
**ステータス**: Accepted
**コンテキスト**: [背景・課題]
**決定内容**: [採用したアプローチ]
**採用理由**: [私が指定した理由 + AIが補足できるトレードオフ]
**却下した代替案**: [AIが代替案とその却下理由を補足する]
**影響**: [この決定がもたらす制約・トレードオフ]

【ガードレール】
- 「採用理由」に私が指定した理由を必ず含めること
- AIが補足する内容は「[AI補足]」と明示すること
```

---

### プロンプト：トレーサビリティマトリクス

```
以下の SWR 一覧と確定した ARC コンポーネント一覧から、
トレーサビリティマトリクスを作成してください。

【SWR 一覧】
[SWR-ID] [SWRタイトル]
（SRS 機能要件・非機能要件の全件）

【ARC 一覧】
[ARC-ID] [コンポーネント名] [担当SWR（私が決めた割り当て）]
（確定した ARC 定義から）

【出力ルール】
| SWR-ID | SWR タイトル | ARC-ID | コンポーネント名 | 備考 |
の形式で出力してください。

以下のケースを「備考」に明記してください：
- 未割り当てのSWR → 「[要確認] 割り当て先なし」
- 複数ARCに分割されたSWR → 「部分割り当て: ARC-xxx + ARC-yyy」
- 私の指定と矛盾する点 → 「[要確認] ○○の理由で矛盾の可能性」
```

## 人間レビューチェックリスト（SAD）

`docs/SWE2/ルール.md` 6章のチェックリストに加えて：

- [ ] コンポーネント定義が自分の意図したスケッチを正確に反映しているか
- [ ] AI が「提案」として追加したコンポーネント・インタフェースを意識的に採否決定したか
- [ ] リソース消費目標の仮定・根拠が実測値・HW仕様と整合しているか
- [ ] ADR の「採用理由」に自分の判断根拠が正確に記載されているか
- [ ] シーケンス図で描かれた呼び出しがすべて IFC-ID に対応しているか
- [ ] 未割り当て SWR がゼロになったか

## SAD 完成後にやること（次工程への橋渡し）

SAD が承認されたら、以下を整理して SDD のガードレールとして使う：

```
【SDD プロンプト用ガードレール（SAD から抽出）】
対象コンポーネント: ARC-xxx [コンポーネント名]
提供するインタフェース: IFC-001(getToken), IFC-002(validateToken)
消費するインタフェース: IFC-010(getUserById from ARC-004)
担当 SWR: SWR-045, SWR-046, SWR-050
禁止事項:
  - 上記以外の外部コンポーネントに依存しないこと
  - 上記以外のインタフェースを公開しないこと
  - SDD 内で新たなコンポーネントを定義しないこと
```

このブロックを SDD の各プロンプトに貼り付けることで、  
AI が SAD の境界を越えた設計を行うことを防ぐ。
