## 必要な Input 資料

| 資料 | 必須/推奨 | 内容 | 入手先 |
|------|----------|------|--------|
| SRS（全要件リスト）| **必須** | SWR-IDと受入基準が含まれたSRS全文 | SWE.1成果物 |
| システム要件仕様書 (CRS/SS) | 推奨 | SWRの上位にあるSYS要件（顧客合意の基準確認用） | 顧客提供 |
| テスト環境仕様 | 推奨 | テスト実施環境（実車 / テストベンチ / 気候条件） | 試験計画書 |
| SWE.5 統合テスト結果 | 推奨 | 前工程で確認済みの事項（重複排除のため） | SWE.5成果物 |

### Input チェックリスト

- [ ] SRSが承認（Approved）済みか
- [ ] SRS全件に受入基準（定量値）が記載されているか
- [ ] テスト実施環境（ターゲット環境）が確定しているか
- [ ] 安全要件（ASIL）に関するテストは特別な手順が必要か確認したか

## プロンプト：SWR全件からのテストケース一括生成

```
以下のSRS要件全件をもとに、ASPICE SWE.6準拠の適格性確認テスト仕様を作成してください。

【プロジェクトコンテキスト】
- プロジェクトID  : [プロジェクトID]
- テスト実施環境  : [例: 実ECU + ターゲット車両 / HILシミュレータ]
- テスト実施条件  : [例: 環境温度 -40〜85℃ / 電源電圧 9〜16V]

【SRS 機能要件一覧】
[SWR-ID] | [要件タイトル] | [受入基準]
SWR-001 | [タイトル] | [定量的な合否判定条件]
SWR-002 | [タイトル] | [定量的な合否判定条件]
（全件貼り付け。受入基準がないものは「受入基準未定義（要確認）」と記載）

【出力ルール】
各SWR-IDに対して1つ以上のテストケースを作成してください：

| テストケースID | 対応SWR | テスト対象機能 | テスト手順 | 入力/操作 | 期待結果 | 合否判定基準 | テスト環境条件 |
|----------------|---------|--------------|-----------|----------|---------|------------|--------------|
| SQT-001        | SWR-001 | [機能名]     | 1. [手順] | [入力]   | [期待値] | [条件]     | [環境]        |

テスト手順は実際の操作担当者が迷わず実行できる粒度（番号付きステップ）で記述してください。
```

## プロンプト：非機能要件のテストケース

```
以下の非機能要件をもとに、システムレベルでの検証テストを設計してください。

【非機能要件一覧（SRSから抜粋）】
[SWR-N01] 性能: [応答時間・スループット・起動時間などの数値要件と受入基準]
[SWR-N02] 安全: [ASIL要件・フェールセーフ動作の要件]
[SWR-N03] セキュリティ: [認証・アクセス制御の要件]
[SWR-N04] 信頼性: [MTBF・可用率の要件]

【テスト環境】: [例: テストベンチ・計測器 (オシロスコープ / CANoe など)]

【出力ルール】
■ 性能テスト
- 負荷条件（最大負荷・平均負荷・最悪ケース）を明示
- 計測方法（計測ポイント・使用ツール・計測回数）を記述
- 合否判定: 受入基準の数値を「〜以内」「〜以上」で明記

■ 安全テスト
- 異常注入方法（電源断 / センサ断線 / 通信エラー）
- フェールセーフ動作の確認手順
- 安全状態への遷移時間の計測方法

■ 信頼性テスト
- 連続動作時間・繰り返し回数
- 観測する劣化指標
- 異常終了基準

テストケース形式は前のプロンプトと同一形式で出力してください。
```

## プロンプト：トレーサビリティマトリクス（SWR → QTS）

```
以下のSWR一覧と作成したテストケース一覧から、
適格性確認テストのトレーサビリティマトリクスを作成してください。

【SWR 全件一覧】
[SWR-ID] [SWRタイトル]
（SRS全件を貼り付け）

【テストケース一覧】
[テストケースID] [対応SWR-ID]
（作成済みSQT-xxx一覧を貼り付け）

【出力ルール】
■ SWR → テストケース マトリクス
| SWR-ID | SWR タイトル | 対応テストID | カバレッジ状況 |
|--------|-------------|------------|--------------|

カバレッジ状況の区分：
- 「テスト済（SQT-xxx）」：テストケースが存在する
- 「未カバー（要追加）」  ：テストケースが存在しない
- 「上流工程で確認済」   ：SWE.5のITSで検証済みで省略

■ 未カバーSWR一覧
未カバーのSWRに対して追加テストケースを提案してください。
```

## プロンプト：回帰テストセット抽出

```
以下の変更内容をもとに、影響を受けるテストケースを特定してください。

【変更内容】
変更したSWR: [SWR-xxx] [変更の概要]
変更したコンポーネント: [ARC-xxx]

【既存テストケース一覧】
[テストケースID] [対応SWR] [対応コンポーネント]
（全QTSテストケース一覧を貼り付け）

【出力ルール】
1. 変更の直接影響を受けるテストケース（対応SWRまたはコンポーネントが一致するもの）
2. 間接影響を受ける可能性があるテストケース（依存コンポーネントを参照するもの）
3. 回帰テスト実施順序の推奨（優先度順）
4. 変更に伴い追加が必要なテストケースの提案
```

## 人間レビューチェックリスト（QTS）

- [ ] 全SWR-IDに対してテストケースが存在するか（未カバーが0件か）
- [ ] テスト手順が実際のテスト担当者（必ずしも設計者でない）が実行できるか
- [ ] 合否判定基準が主観的判断を要せず、明確にOK/NGを判定できるか
- [ ] 性能テストの計測条件が実際のターゲット環境と一致するか
- [ ] ASIL要件に関するテストが安全計画の要求を満たしているか
- [ ] 顧客・認証機関に提出する形式・内容として適切か
