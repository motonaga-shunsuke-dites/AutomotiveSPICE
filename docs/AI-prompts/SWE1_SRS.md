## 必要な Input 資料

| 資料 | 必須/推奨 | 内容 | 入手先 |
|------|----------|------|--------|
| システム要件仕様書 (CRS/SS) | **必須** | 顧客・上位からのシステムレベル要件。SYS-xxx IDが付いていることが望ましい | 顧客提供 or 上流チーム |
| システムアーキテクチャ概要 | **必須** | ソフトウェアがどのHW・システムの中で動くかの全体像 | 上流チーム |
| インタフェース要件定義 | 推奨 | 外部システム・HWとの接続仕様（CAN/LIN/Ethernetなど） | HW設計チーム |
| 既存SRSの類似ドキュメント | 推奨 | 過去プロジェクトの参考。構成の参考になる | 社内ドキュメント |

### Input チェックリスト（作業開始前に確認）

- [ ] CRS/SSに一意のIDが付いているか
- [ ] ソフトウェアの対象範囲（スコープ）が明確か
- [ ] ターゲットHW・OS・通信プロトコルが確定しているか
- [ ] 安全要件（ASIL）が定義されているか

## プロンプト：章1〜3（目的・参照・用語）

```
以下のコンテキストとシステム要件をもとに、ASPICE SWE.1準拠のSRSの
「章1 目的・スコープ」「章2 参照文書」「章3 用語定義」を作成してください。

【プロジェクトコンテキスト】
- プロジェクト名    : [プロジェクト名]
- プロジェクトID    : [プロジェクトID]
- 対象ソフトウェア  : [ソフトウェア名]
- ターゲット環境    : [MCU / OS / ミドルウェア]
- プログラミング言語: [言語・準拠規格]
- 安全要件          : [ASIL レベル]

【システム要件仕様書（抜粋）】
[CRS/SSの内容をここに貼り付け]

【出力形式】
SRS_template.md の章1〜3の構造に従い、Markdown形式で出力してください。
スコープには「対象」と「対象外」を明確に分けてください。
用語定義は本書で使う略語・専門用語を網羅してください。
```

## プロンプト：章5 機能要件（最重要）

```
以下のシステム要件をSWE.1準拠のソフトウェア機能要件に変換してください。

【プロジェクトコンテキスト】
- プロジェクトID    : [プロジェクトID]
- 対象ソフトウェア  : [ソフトウェア名]

【変換するシステム要件】
[SYS-001] [システム要件の内容]
[SYS-002] [システム要件の内容]
（必要な件数分貼り付ける）

【出力ルール】
各要件を以下のフィールドを持つブロックで出力してください：
- ID: SWR-xxx（3桁連番）
- タイトル: 30字以内・主語＋動詞
- 説明: 「システムは〜しなければならない (shall)」形式
- 優先度: Must / Should / Nice-to-have
- 根拠: 対応する SYS-ID
- 受入基準: 定量的な合否判定条件（数値・閾値を含む）
- ステータス: Draft

【出力制約】
- 「速やかに」「適切に」「十分に」などの曖昧表現を使わない
- 受入基準には必ず数値（時間・回数・割合など）を含める
- 1 SWR = 1 要件。複数の要件を1IDに混在させない
- 実装方法(HOW)でなく要件(WHAT)を記述する
- 1つの SYS 要件から複数の SWR を派生させてよい
```

## プロンプト：章6 非機能要件

```
以下のプロジェクト制約をもとに、SWE.1準拠の非機能要件を作成してください。

【プロジェクトコンテキスト】
- プロジェクトID    : [プロジェクトID]
- ターゲット環境    : [MCU / OS]
- 安全要件          : [ASIL レベル]
- 既知の制約        : [応答時間・メモリ・通信速度などの数値制約]

【上位で定義された非機能要件（あれば）】
[システムレベルの非機能要件を貼り付け]

【出力ルール】
以下の種別ごとに、ID・要件説明・目標値・根拠のテーブルを作成してください：
- 性能要件（応答時間・スループット・起動時間）
- 安全要件（ASIL・フェールセーフ・診断機能）
- セキュリティ要件（認証・暗号・アクセス制御）
- 信頼性要件（MTBF・可用率・エラー検出率）
- 保守性要件（診断ログ・OTA対応・テスト容易性）

ID採番: SWR-N01 から連番。
目標値には必ず単位と数値を含める。
```

## プロンプト：章7 インタフェース要件

```
以下のシステム構成をもとに、SWE.1準拠のインタフェース要件を作成してください。

【プロジェクトコンテキスト】
- 対象ソフトウェア  : [ソフトウェア名]
- ターゲット環境    : [MCU / OS]

【接続するシステム・HWの情報】
[インタフェース要件定義書の内容、または以下の形式で記述]
- HWインタフェース: [CAN / LIN / SPI / I2C など + 通信速度・プロトコル]
- 外部システム    : [接続先システム名 + データ形式]
- ユーザインタフェース: [操作パネル / ディスプレイ / センサ など]

【出力ルール】
3種類（HW・外部システム・ユーザ）のテーブルを作成してください。
各インタフェースに SWR-Ixx の ID を付与してください。
通信方式・プロトコル・データ形式・タイミング制約を明記してください。
```

## プロンプト：章9 トレーサビリティマトリクス

```
以下の SYS 要件と SWR の一覧から、トレーサビリティマトリクスを作成してください。

【SYS 要件一覧】
[SYS-ID] [SYS要件タイトル]
（一覧を貼り付け）

【SWR 一覧】
[SWR-ID] [SWRタイトル] [根拠SYS-ID]
（一覧を貼り付け）

【出力ルール】
| SYS-ID | SYS要件タイトル | SWR-ID | SWRタイトル | 備考 |
の形式で出力してください。

以下のケースを「備考」に明記してください：
- 1つのSYS要件が複数のSWRに分割された場合: 「部分割り当て」
- SYS要件に対応するSWRが存在しない場合: 「未割り当て（要確認）」
- SWRにSYS-IDが存在しない場合: 「追加根拠必要」
```

## 人間レビューチェックリスト（SRS）

`docs/SWE1/ルール.md` 6章のチェックリストに加えて：

- [ ] SYS要件の抜け漏れがないか（すべてのSYS-IDにSWRが存在するか）
- [ ] 受入基準がテスト環境で実際に検証可能か
- [ ] 安全要件（ASIL）の記述が組織の安全計画と整合しているか
- [ ] インタフェース要件がHW設計チームと合意された仕様と一致するか
- [ ] 「対象外」の記述が明確で関係者の合意を得られているか
