## SWE.3 における AI の役割と境界

SAD が「ガードレール」として機能する。AI は SAD に定義された **1 コンポーネントの内側だけ** を設計する。

```
SAD で定義済み（変更禁止）    │  SDD で AI が設計する範囲
─────────────────────────────────────────────────────────────
ARC-003 AuthManager           │  クラス構造・メソッド詳細
  提供: IFC-001(getToken)     │  前提条件・事後条件
  提供: IFC-002(validateToken)│  エラーコード定義
  消費: IFC-010(getUserById)  │  シーケンス図（既存IFCの範囲内）
  担当: SWR-045, 046, 050     │  状態遷移設計
                              │  ← SAD の外側は触らない
```

AI への必須ガードレール（全 SDD プロンプトに含める）:
```
【SDD ガードレール】
- このSDD は ARC-xxx のみを扱うこと
- SAD に定義された以外のコンポーネントを追加しないこと
- 公開インタフェースは IFC-xxx で定義されたものに限定すること
- 上記以外のコンポーネントに依存しないこと
- 変更提案がある場合は「[SAD変更提案]」と明示し、私の確認を求めること
```

## 必要な Input 資料

| 資料 | 必須/推奨 | 内容 | 入手先 |
|------|----------|------|--------|
| **SAD の対象 ARC 章のみ**（全体不要）| **必須** | ARC-IDとその責務・インタフェース・担当SWR | SWE.2成果物 |
| **対象 ARC の IFC 定義のみ**（全体不要）| **必須** | 提供・消費するAPI仕様（型・引数・戻り値） | IFD |
| 担当 SWR の内容（SRSから抜粋） | **必須** | ARC-IDに割り当てられたSWRの詳細 | SRS |
| コーディング規約 | 推奨 | MISRA-C・命名規則・禁止パターン | docs/SWE3/テンプレート/ |
| 既存コード（あれば） | 推奨 | 流用・参考にする既存実装 | ソースリポジトリ |

> **重要**: プロンプトに SAD の全体を貼らない。対象 ARC の章のみを渡す。  
> AI が他コンポーネントを参照・変更する余地を与えないことがガードレールになる。

### Input チェックリスト（作業開始前に確認）

- [ ] SAD が承認済みか
- [ ] 対象 ARC の責務・インタフェース・担当 SWR が SAD に明記されているか
- [ ] 「SDD ガードレールブロック」（上記フォーマット）を準備したか
- [ ] 使用するデータ型・列挙型が他コンポーネントと合意されているか

> **原則**: SDD は 1 ARC-ID ごとに 1 枚作成する。

## プロンプト：章4〜5 クラス図・モジュール構造図（最重要）

```
以下のコンポーネント仕様をもとに、
クラス図・モジュール構造図をPlantUML形式で作成してください。

【コンポーネント仕様（SADから抜粋）】
コンポーネント名: [名前]
責務: [責務の説明]
提供インタフェース:
  - [関数名(引数): 戻り値型]
消費インタフェース:
  - [依存する外部API]
担当SWR: [SWR-IDリスト]

【担当SWRの詳細】
[各SWR-IDの要件説明をSRSから貼り付け]

【出力ルール】
■ モジュール構造図 (@startuml package記法)
- このコンポーネントのファイル構成（.h / .c の関係）
- 外部依存するモジュールとの依存関係

■ クラス図 (@startuml class記法)
- 主要な構造体・クラス
- メンバ変数（型・役割）
- メソッド（公開API＋内部関数、-/+で区別）
- 継承・集約・依存関係

【制約】
- ヘッダファイルで公開する型と内部実装を明確に分けること
- C言語の場合は typedef struct / enum を使うこと
```

```

【対象API】

【インタフェース定義（IFDから抜粋）】
引数: [型 名前, ... ]
戻り値: [型・意味]
呼び出しタイミング: [定期 / イベント駆動]

【出力ルール】
以下のフィールドを含む関数仕様を作成してください：
- 関数ID: UNT-xxx
- 関数名・シグネチャ
- 概要（2〜3行）
- 前提条件 (Precondition): 呼び出し前に満たされるべき状態
- 事後条件 (Postcondition): 正常完了後の保証
- 引数詳細: 型・範囲・単位・NULLの可否
- 戻り値詳細: 各戻り値コードの意味
- エラー処理: 各エラーケースでの動作
- 制約・注意事項: スレッドセーフ性・再入可能性・副作用

（このプロンプトをAPIごとに繰り返す）
```

## プロンプト：章8 状態遷移設計

```
以下のコンポーネント仕様をもとに、状態遷移設計を作成してください。

【コンポーネント名】: [コンポーネント名]
【担当SWR】: [SWR-IDリスト]
【関連するSWR内容】: [状態に関連するSWRの説明を貼り付け]

【出力ルール】
■ 状態遷移図 (@startuml state記法)
- すべての状態（初期状態・中間状態・終了状態）
- 各遷移のトリガー条件（イベント名[ガード条件]/アクション）
- フェールセーフ状態への遷移

■ 状態遷移テーブル
| 現在の状態 | イベント | ガード条件 | 次の状態 | アクション |
の形式で全遷移を網羅してください。

【制約】
- 「未定義状態」への遷移は絶対に作らない（デフォルト遷移にエラー処理を含める）
- フェールセーフ状態からは通常状態へ直接遷移できないこと（リセット必要）
```

## 人間レビューチェックリスト（SDD）

`docs/SWE3/ルール.md` のチェックリストに加えて：

- [ ] 関数仕様の前提条件が実際の呼び出し元コードで保証できるか
- [ ] エラーコードが他コンポーネントと重複していないか
- [ ] 状態遷移テーブルにすべての状態×イベントの組み合わせが記述されているか（抜けは「-（遷移なし）」を明記）
- [ ] スレッドセーフ性・再入可能性が要件（RTOS使用）と整合しているか
- [ ] PlantUMLのシーケンス図がコンポーネントの実際のAPI名と一致しているか
- [ ] DB設計がターゲット環境の制約（Flash書き込み寿命・アドレスアライメント）を満たしているか
