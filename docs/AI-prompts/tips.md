# AI作業 Tips・よくある失敗パターン

---

## AI 設計の大原則

**「人間が構造を定義し、AI が詳細を記述する」**

AIは「局所最適が得意、全体設計が苦手」という特性を持つ（Spring Boot 設計崩壊の防止に関する記事より）。
ASPICE の工程構造はこの特性と相性が良く、適切に使えば品質が安定する。

```
×  AI が全体設計（コンポーネント分割・責務設計）を行う
✓  人間がスケッチ → AI が1コンポーネント単位で詳細化
```

---

## 各工程の AI 活用の位置づけ

```
SWE.1（SRS） ─── AI が初稿生成 ── 局所的作業（要件変換）のためAI向き
SWE.2（SAD） ─── 人間が先にスケッチ → AI が清書・ギャップ指摘
              ↑ ここだけ人間主導。コンポーネント境界の決定は全体設計
SWE.3（SDD） ─── AI が生成（1 ARC-ID ずつ）── SAD が境界を定めるため安全
SWE.4（UTS） ─── AI が生成 ── 関数仕様 → テスト生成は規則的
SWE.5（ITS） ─── AI が生成 ── インタフェース定義 → テスト生成は局所的
SWE.6（QTS） ─── AI が生成 ── SWR → テストは1対1展開
```

## ガードレールの仕組み

### SAD が .cursorrules として機能する

コードベースで `.cursorrules` に「Controller は Service のみ参照可」と書くように、
ASPICE では SAD の ARC 定義が SDD への制約になる。

```
SAD で定義したもの:
  ARC-003 AuthManager
    提供インタフェース: IFC-001, IFC-002
    消費インタフェース: IFC-010 (ARC-004 UserRepository から)
    担当 SWR: SWR-045, 046, 050

↓ SDD プロンプトに毎回このブロックを貼る

【SDD ガードレール】
- このSDD は ARC-003 のみを扱うこと
- 公開インタフェースは IFC-001, IFC-002 のみ
- ARC-004 以外のコンポーネントに依存しないこと
- SAD に存在しない依存・インタフェースを追加しないこと
- 変更提案は [SAD変更提案] と明示して確認を求めること
```

これにより、AI が「やりすぎる」（勝手にコンポーネントを追加する等）を防ぐ。

## 生産性を上げるパターン

### 1. 「叩き台生成 → 章ごと修正」の反復

全章を一気に作らせると後半の質が落ちる。

```
[第1回] 「章1〜4を作成して」
[第2回] 「章4を修正。[具体的な修正内容]。そのまま章5を続けて」
[第3回] 「SWR-003の受入基準を『20ms以内』に変更して。章6に進んで」
```

### 2. 前回出力を引き継ぐ

```
前回確定した章4を前提として、章5（コンポーネント構成）を作成してください。

【章4（確定済み）】
（確定した章4のテキストをそのまま貼り付け）
```

### 3. 「修正差分だけ」を依頼する

```
SWR-005 の受入基準だけを変更してください。
変更前: 「適切な時間内」
変更後: 「100ms 以内（CPU使用率70%以下の条件下）」
他は変更しないでください。
```

### 4. SAD スケッチから SDD ガードレールを自動生成させる

```
以下の SAD の ARC 定義から、各 ARC の「SDD ガードレールブロック」を
生成してください。

【SAD ARC 一覧】
（SAD の章5をそのまま貼り付け）

各 ARC について以下の形式で出力してください：
=== ARC-xxx [コンポーネント名] ===
【SDD ガードレール】
- このSDD は ARC-xxx のみを扱うこと
- 公開インタフェース: [IFC-IDのリスト]
- 消費インタフェース: [IFC-IDのリスト]
- 担当 SWR: [SWR-IDのリスト]
- SAD に存在しない依存・インタフェースを追加しないこと
```

生成したブロックを各 SDD プロンプトのヘッダに貼り付けて使う。

## よくある失敗パターンと対策

### ❌ 失敗1: SAD なしに SDD を依頼する（全体設計をAIに任せる）

```
× 「認証機能の詳細設計書を作ってください」
```
→ AI が架空のコンポーネントを生成する。後から SAD との整合修正が大変になる。

```
✓ 「ARC-003 AuthManager（SAD章5より）と担当SWR（SWR-045〜050）を
   渡した上で、SDD の章4〜6を作成してください。
   SAD に定義されていない依存・インタフェースを追加しないこと」
```

---

### ❌ 失敗2: SAD の生成を AI に全部任せる

```
× 「SRSをもとにアーキテクチャを設計してください」
```
→ AI がコンポーネント分割を決める。実装チームの能力・既存資産・HW制約を考慮できない。

```
✓ 手順:
  1. 人間が「コンポーネント名・責務・接続先」のメモを作る（10分）
  2. AI に「このスケッチを SAD 形式に清書して。ギャップを指摘して」と依頼
  3. AI の指摘を受けてスケッチを修正（人間が判断）
  4. 確定後に詳細化を依頼
```

---

### ❌ 失敗3: 数値をそのまま信用する

AI は「それらしい数値」を生成する。受入基準・リソース配分・タイミング値は必ず元仕様と突き合わせる。

```
□ 受入基準の数値は SYS 要件から来ているか？
□ リソース配分の合計は HW 仕様を超えていないか？
□ タイミング値は HW 仕様書の実測値と一致しているか？
```

---

### ❌ 失敗4: ID の採番を AI に任せる

AI は ID をランダムに飛ばしたり重複させたりする。

→ **ID の採番は人間が管理する**。「SWR-001 から SWR-010 の範囲で作成」と明示する。

---

### ❌ 失敗5: 完全なドキュメントを一度に作らせる

→ **章単位で生成・確認・承認してから次章へ進む**。後半ほど品質が落ちる。

## コピー用：共通プロンプトブロック

```
【プロジェクトコンテキスト】
- プロジェクト名    : 
- プロジェクトID    : 
- 対象ソフトウェア  : 
- ターゲット環境    : 
- プログラミング言語: 
- 安全要件（ASIL） : 
- プロジェクト制約  : 

【ガードレール（毎回必須）】
- SAD に定義されていないコンポーネント・インタフェースを追加しないこと
- 変更提案がある場合は [提案] または [SAD変更提案] と明示して確認を求めること
- 「速やかに」「適切に」「十分に」などの曖昧表現を使わないこと
- 要件には shall を使い、定量値を必ず含めること
- 1 ID = 1 要件
- PlantUML のコードブロックは ```plantuml で囲むこと
- Markdown テーブルは必ずヘッダ行と区切り行を含むこと
```
