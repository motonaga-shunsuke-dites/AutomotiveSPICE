## 必要な Input 資料

| 資料 | 必須/推奨 | 内容 | 入手先 |
|------|----------|------|--------|
| SAD（インタフェース設計章） | **必須** | IFC-IDごとのAPI仕様・通信方式・エラー処理 | SWE.2成果物 |
| IFD（インタフェース設計書） | **必須** | コンポーネント間・外部システムとのAPI詳細 | SWE.2成果物 |
| SRS（機能要件・非機能要件） | **必須** | 統合レベルで確認すべき要件 | SWE.1成果物 |
| 各コンポーネントのSDD | 推奨 | 統合ポイントのエラー処理仕様 | SWE.3成果物 |
| 統合計画 | 推奨 | 統合順序・ビッグバン vs インクリメンタル | プロジェクト計画書 |

### Input チェックリスト

- [ ] 全コンポーネントのSWE.4（ユニットテスト）が完了しているか
- [ ] 統合順序（どのコンポーネントから統合するか）が決まっているか
- [ ] テスト環境（実機 / HIL / シミュレータ）が確定しているか
- [ ] 外部システムのスタブ・シミュレータが使用可能か

## プロンプト：統合テストシナリオ設計（インタフェースペアごとに繰り返す）

```
以下のインタフェース仕様をもとに、ASPICE SWE.5準拠の結合テスト仕様を作成してください。

【プロジェクトコンテキスト】
- プロジェクトID  : [プロジェクトID]
- 統合対象        : [コンポーネントAとコンポーネントBの統合]
- テスト環境      : [例: HILシミュレータ / 実ECU + CANoe]

【統合するインタフェース（SADのIFDから抜粋）】
インタフェースID: [IFC-xxx]
提供コンポーネント: [ARC-xxx] の [関数名(引数): 戻り値]
消費コンポーネント: [ARC-yyy]
呼び出しタイミング: [定期10ms / CAN受信イベント駆動]
エラー処理: [タイムアウト → ○○ / 異常値 → ○○]

【関連SWR】: [SWR-xxx]

【出力ルール】
以下のシナリオタイプごとにテストケースを作成してください：

1. 正常系
   - 各引数の代表値での正常動作確認
   - 連続呼び出し・周期呼び出しの整合性

2. インタフェース異常系
   - タイムアウト（提供側が規定時間内に応答しない）
   - 不正値返却（提供側がエラーコードを返す）
   - データ破損（チェックサムエラー / 範囲外値）

3. 並行アクセス（RTOSの場合）
   - 複数タスクからの同時アクセス
   - 割り込みとの競合

4. 境界・負荷
   - 最大呼び出し頻度での動作
   - リソース枯渇時の動作

各テストケースを以下の形式で出力してください：
| テストケースID | 対応IFC | 対応SWR | テスト観点 | 前提状態 | 操作・入力 | 期待結果 | 合否判定基準 |
|----------------|---------|---------|-----------|---------|-----------|---------|------------|
```

## プロンプト：外部システム統合テスト

```
以下の外部システムとのインタフェースについて、統合テスト仕様を作成してください。

【外部システム情報（IFDから抜粋）】
インタフェースID: [IFC-xxx]
相手システム名  : [例: ECU-B / ゲートウェイECU / 診断ツール]
通信方式        : [例: CAN / LIN / Ethernet / SPI]
通信プロトコル  : [例: UDS / CANopen / カスタム]
メッセージ/フレーム定義: [ID / DLC / 信号定義]
タイミング制約  : [例: 周期10ms / 応答タイムアウト100ms]

【テスト環境】
- 外部システムのシミュレーション方法: [例: CANoe / 実機]
- テスト用ツール: [例: Vector CANalyzer / ETAS INCA]

【出力ルール】
1. 正常通信シナリオ（メッセージ送受信・タイミング確認）
2. 通信エラーシナリオ（断線 / バスオフ / タイムアウト）
3. プロトコルエラーシナリオ（不正フレーム / チェックサム不一致）
4. 実環境互換性テスト（ベンダーの実機との相互接続確認項目）

テストケース形式は前のプロンプトと同一形式で出力してください。
```

## プロンプト：エンド・ツー・エンド（E2E）シナリオテスト

```
以下の主要ユースケースについて、E2E統合テストシナリオを作成してください。

【ユースケース】: [例: ○○機能の正常動作 / 起動〜停止シーケンス / エラー回復フロー]
【関連SWR】: [SWR-xxx, SWR-yyy]
【関与するコンポーネント】: [ARC-001 → ARC-002 → ARC-003 の順に処理]

【シーケンス図（SADから抜粋）】
（関連するシーケンス図またはその説明を貼り付け）

【出力ルール】
1. テストシナリオの全体フロー（番号付きステップで記述）
2. 各ステップの期待値（中間値も含む）
3. 観測ポイント（どの値をどのツールで観測するか）
4. 合否判定基準（定量的な合否条件）
5. 異常注入バリエーション（各ステップで異常を発生させた場合のバリエーション）
```

## プロンプト：テストカバレッジ確認

```
以下のSADのインタフェース一覧と作成したテストケースを照合し、
カバレッジを分析してください。

【インタフェース全件一覧（SADから）】
[IFC-ID] [インタフェース名] [提供コンポーネント → 消費コンポーネント]
（IFD全件を貼り付け）

【作成済みテストケース一覧】
[テストケースID] [対応IFC]
（作成済みUTCの一覧を貼り付け）

【出力ルール】
1. 各IFCに対してカバーするテストケースIDを列挙
2. テストケースが存在しないIFCを「未カバー」として特定
3. 未カバーIFCに対する追加テストケースを提案
4. SWR対応テーブル（各SWRを検証するテストケースID）
```

## 人間レビューチェックリスト（ITS）

- [ ] 全IFC-IDに対してテストケースが存在するか
- [ ] 通信タイムアウト値がSDD・IFDの規定値と一致しているか
- [ ] 外部システムとのテストで実機との接続確認が含まれているか
- [ ] E2Eシナリオが重要なSWR（特にASIL要件）を網羅しているか
- [ ] テスト環境の制約（CAN負荷・HIL応答時間）がテストケースの前提と一致するか
- [ ] 並行アクセステストのタスク優先度設定が実際のOSコンフィグと一致するか
