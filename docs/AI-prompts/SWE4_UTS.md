## 必要な Input 資料

| 資料 | 必須/推奨 | 内容 | 入手先 |
|------|----------|------|--------|
| SDD（対象モジュールの関数仕様章） | **必須** | UNT-IDごとの前提条件・事後条件・エラー処理 | SWE.3成果物 |
| 状態遷移テーブル（SDDから） | **必須** | 全状態×イベントの組み合わせ | SDD章8 |
| コーディング規約 | 推奨 | 禁止API・MISRA-C準拠要件 | docs/SWE3/テンプレート/ |
| テストフレームワーク仕様 | 推奨 | CUnit / Unity / GoogleTestのAPI | ツール仕様書 |

### Input チェックリスト

- [ ] SDDが承認済みか（少なくともReview済み）
- [ ] テストフレームワーク・テスト環境が決定しているか
- [ ] テストカバレッジ目標（C0/C1/MC/DC）が定義されているか

## プロンプト：テストケース設計（関数ごとに繰り返す）

```
以下の関数仕様をもとに、ASPICE SWE.4準拠のユニットテスト仕様を作成してください。

【プロジェクトコンテキスト】
- プロジェクトID       : [プロジェクトID]
- コンポーネント名     : [コンポーネント名]
- テストフレームワーク : [例: Unity / CUnit / gtest]
- カバレッジ目標       : [例: C1（分岐網羅）100% / MC/DC]

【対象関数仕様（SDDから抜粋）】
関数ID  : [UNT-xxx]
関数名  : [関数名(引数型): 戻り値型]
前提条件: [呼び出し前に保証される条件]
事後条件: [正常終了後の状態]
引数詳細:
  - [引数名]: 型=[型], 範囲=[min〜max], 単位=[単位]
戻り値  :
  - OK(0): [意味]
  - ERR_xx(-1): [意味]
エラー処理:
  - [異常ケース] → [動作]

【出力ルール】
以下の手法でテストケースを網羅的に作成してください：
1. 同値分割（有効同値・無効同値）
2. 境界値分析（最小値・最小値+1・最大値-1・最大値・境界外）
3. エラーケース（各エラー条件を個別にテスト）
4. 状態依存（前提条件を変えた場合の動作）

各テストケースを以下の形式で出力してください：
| テストケースID | 対応UNT | テスト観点 | 前提条件 | 入力 | 期待結果 | 備考 |
|----------------|---------|-----------|---------|------|---------|------|
| UTC-xxx-001    | UNT-xxx | [観点]   | [前提] | [入力値] | [期待値] | |

（このプロンプトを関数ごとに繰り返す）
```

## プロンプト：状態遷移テスト

```
以下の状態遷移テーブルをもとに、状態遷移テストケースを作成してください。

【コンポーネント名】: [コンポーネント名]

【状態遷移テーブル（SDDから抜粋）】
| 現在の状態 | イベント | ガード条件 | 次の状態 | アクション |
（SDDの状態遷移テーブルを貼り付け）

【出力ルール】
以下の方針でテストケースを作成してください：
1. 各遷移（行）に対して1つ以上のテストケース
2. 「-（遷移なし）」の組み合わせ → 無効遷移テスト（何も起きないことを確認）
3. スヌークカバレッジ（全状態を1回以上通過する最小シナリオ）

出力形式：
| テストケースID | 初期状態 | 入力イベント | ガード条件 | 期待遷移先 | 期待アクション |
|----------------|---------|------------|-----------|-----------|--------------|
```

## プロンプト：テストスタブ・モック設計

```
以下の依存関係をもとに、ユニットテスト用のスタブ/モック仕様を作成してください。

【テスト対象コンポーネント】: [コンポーネント名]
【依存コンポーネント（消費するAPI）】:
  - [ARC-xxx] の [関数名(引数): 戻り値]
  - [ARC-yyy] の [関数名(引数): 戻り値]

【テストフレームワーク】: [例: Unity + CMock / fff]

【出力ルール】
各依存APIについて：
1. スタブ関数の定義（戻り値を固定値で返す基本スタブ）
2. モック制御パラメータ（呼び出し回数・引数確認・戻り値の差し替え）
3. テストケースごとの設定例（呼び出し前に何をセットアップするか）
```

## プロンプト：カバレッジ分析

```
以下のテストケース一覧と関数実装をもとに、カバレッジ分析を行ってください。

【テストケース一覧】
（UTC-xxx-001〜 の一覧を貼り付け）

【関数実装（疑似コードまたは実際のCコード）】
[コードを貼り付け]

【カバレッジ目標】: [C1 / MC/DC]

【出力ルール】
1. 各分岐（if/else/switch）に対してカバーするテストケースIDを列挙
2. 未カバーの分岐を特定し、追加テストケースを提案してください
3. MC/DCの場合: 各条件の独立影響ペアとカバー状況を表で示してください
```

## 人間レビューチェックリスト（UTS）

- [ ] 全UNT-IDに対してテストケースが存在するか
- [ ] 境界値が要件の数値と一致しているか（SDDと数値を突き合わせる）
- [ ] 状態遷移テーブルの全行にテストケースが対応しているか
- [ ] スタブの戻り値設定でテスト対象の全エラーパスが通るか
- [ ] テスト環境（HW依存部分）で実行可能か
- [ ] カバレッジ目標を達成できるテストケース数か
