# AI ドキュメント作成ガイド

## 基本方針：「人間が構造を定義し、AIが詳細を記述する」

AIは「**局所最適が得意、全体設計が苦手**」（参考: AIを活用した開発における設計崩壊の防止）。

この特性を ASPICE の工程構造に当てはめると、以下の分担が自然に決まる。

| 工程 | 主担当 | 理由 |
|------|--------|------|
| SWE.1 SRS | **AI（人間が確認）** | 上位要件 → SW要件への変換は局所的作業 |
| SWE.2 SAD | **人間（AIが補助）** | コンポーネント境界・責務の決定は全体設計 |
| SWE.3 SDD | **AI（人間が確認）** | 1コンポーネント = 局所文脈。AIの得意領域 |
| SWE.4 UTS | **AI（人間が確認）** | 関数仕様 → テストケース生成は規則的作業 |
| SWE.5 ITS | **AI（人間が確認）** | インタフェース定義 → テストは局所的作業 |
| SWE.6 QTS | **AI（人間が確認）** | SWR → テストケースは1対1展開 |

### SAD を「ガードレール」として機能させる

SAD（SWE.2）に定義したコンポーネントの**外側**を、AI が設計しないようにする。  
これが記事における `.cursorrules` / `CLAUDE.md` のルールに相当する。

```
SAD = コンポーネント境界の定義
  ↓
SDD プロンプトに「ARC-xxx の範囲内のみ設計すること」を必ず明示
  ↓
AI は1コンポーネントの局所設計に集中 → 品質が安定する
```

## 工程別の進め方

### SWE.2 SAD ：人間が先にスケッチを描く

**AIに先に「設計して」と依頼してはいけない。**  
以下の順序で作業する：

```
Step1（人間）: コンポーネントをホワイトボードに列挙する
              「認証、通信、制御、ログ」など名前と責務だけでよい

Step2（人間）: コンポーネント間の依存関係・主要インタフェースをメモする

Step3（AI）  : スケッチをSAD形式に清書・詳細化・PlantUML化・ギャップ指摘

Step4（人間）: ギャップ指摘を受けて設計を修正・確定
```

AIに渡すときは「設計して」ではなく「**以下のスケッチをSAD形式に清書して**」と依頼する。

### SWE.3 SDD 以降：AI 主導で局所設計

SAD が確定したら、1 ARC-ID ごとに AI に依頼する。  
プロンプトには必ず「このSDD は ARC-xxx のみを扱い、SAD に定義されていないコンポーネントを追加しないこと」を含める。

## 共通プロンプト構造

毎回のプロンプトは以下の構造で作る：

```
【プロジェクトコンテキスト】
- プロジェクト名    : 
- プロジェクトID    : 
- 対象ソフトウェア  : 
- ターゲット環境    : 
- プログラミング言語: 
- 安全要件（ASIL） : 
- プロジェクト制約  : 

【ガードレール（毎回必須）】
- SAD に定義されていないコンポーネント・インタフェースを追加しないこと
- 「速やかに」「適切に」「十分に」などの曖昧表現を使わないこと
- 要件には shall を使い、定量値を必ず含めること
- 1 ID = 1 要件
- 実装方法(HOW)でなく要件(WHAT)を記述すること

【依頼内容】
（各工程のプロンプトをここに続ける）
```

## 工程別 Input 定義

| 工程 | 必須 Input | 推奨 Input | 作成ドキュメント |
|------|-----------|-----------|-----------------|
| SWE.1 | システム要件仕様書 (CRS/SS) | HW仕様書、インタフェース要件 | SRS, IRS, TM |
| SWE.2 | **人間が描いたコンポーネントスケッチ** + SRS | システムアーキテクチャ、既存設計 | SAD, IFD, TM |
| SWE.3 | **SAD の対象 ARC 章**（全体ではなく対象のみ）+ 関連 SWR | 既存コード | SDD, TM |
| SWE.4 | SDD の対象関数仕様章 | コーディング規約 | UTS |
| SWE.5 | SAD のインタフェース定義 + SRS | 統合計画 | ITS |
| SWE.6 | SRS 全件（SWR-ID + 受入基準） | テスト環境仕様 | QTS |

## 人間が必ず担う判断

| 判断事項 | なぜAIに任せられないか |
|---|---|
| コンポーネント境界の設定（SAD） | プロジェクト全体の変更影響範囲を判断する必要がある |
| コンポーネントの責務割り当て（SAD） | チームの実装能力・既存資産を考慮する必要がある |
| ASIL 判定・安全要件の妥当性 | 機能安全規格の解釈はドメイン専門家が行う |
| トレーサビリティの完全性確認 | ID マッピングミスを見逃す |
| インタフェースの物理的実現可能性 | HW 仕様・通信タイミングをAIは知らない |
| 数値・単位の正確性 | それらしい数値を生成することがある |

### [TODO-AI-001] 工程別 Claude Project（カスタムエージェント）の作成

**概要**: 各SWE工程専用の Claude Project を作成し、以下をシステムプロンプトに組み込む。
- 対象工程の `ルール.md`（フォーマット・禁止事項）
- 対象工程のテンプレート
- SAD（確定済みコンポーネント定義）をガードレールとして登録
- プロジェクトコンテキスト（固定）

**期待効果**:
- コンテキストを毎回貼る手間がなくなる
- 「SAD にないコンポーネントを追加する」違反を構造的に防ぐ
- 工程間の整合性チェックが自動化できる

**実現方法の候補**:
1. Claude.ai の Projects 機能（Knowledge に `ルール.md` + 確定済み SAD を登録）
2. ASPICE Doc Tool にプロンプト生成機能を追加（ルール + SAD + Input を自動結合してクリップボードにコピー）
3. カスタム API エージェントとして実装し、整合性チェックも組み込む

**優先度**: 中（手作業フローが定着してから）

### [TODO-AI-002] ドキュメント間整合性チェック機能

**概要**: 「SDD に ARC-ID に存在しないコンポーネントが書かれていないか」など、  
ArchUnit 相当のドキュメント版整合性検証を AI に自動実施させる。

**チェック候補**:
- SDD のすべてのインタフェースが IFD（IFC-ID）に存在するか
- SDD が参照する SWR が SRS に存在するか
- SRS のすべての SWR が少なくとも1つの ARC に割り当てられているか
- QTS のすべてのテストケースが対応する SWR を持つか

**優先度**: 高（工程が増えるほど手作業チェックが困難になるため）
