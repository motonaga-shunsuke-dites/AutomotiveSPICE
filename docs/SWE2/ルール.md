# SWE.2 ソフトウェアアーキテクチャ設計 — ドキュメントルール

## 1. プロセス概要

### 目的

SRS に定義されたソフトウェア要件を実現するための**アーキテクチャ構造を設計**する。  
ソフトウェアをコンポーネントに分割し、コンポーネント間のインターフェース・動的ふるまい・  
リソース消費目標を定義することで、SWE.3（詳細設計）の基盤を提供する。

### 位置づけ（入出力）

```
入力:
  - ソフトウェア要件仕様書 (SRS)
  - インターフェース要件定義書

出力:
  - ソフトウェアアーキテクチャ設計書 (SAD)
  - インターフェース設計書（SW コンポーネント間）
  - トレーサビリティマトリクス（SWR → ARC）

後続:
  - SWE.3（SAD を入力として詳細設計へ）
  - SWE.5（統合戦略の基礎として参照）
```

## 4. ドキュメント記述ルール

### 4.1 ソフトウェアアーキテクチャ設計書 (SAD)

#### 4.1.1 必須構成

| 章番号 | 章タイトル           | 必須記載内容                                           |
|--------|----------------------|--------------------------------------------------------|
| 1      | 目的・スコープ       | 本 SAD の適用範囲・設計方針の概要                      |
| 2      | 参照文書             | SRS・標準規格・設計パターンのリスト                    |
| 3      | 用語定義             | アーキテクチャ固有の用語・略語                         |
| 4      | アーキテクチャ概要   | 全体構成の説明・設計思想・採用パターン                 |
| 5      | コンポーネント構成   | 各コンポーネントの責務・依存関係・階層構造              |
| 6      | インターフェース設計 | コンポーネント間 API・データ形式・通信方式              |
| 7      | 動的ふるまい         | シーケンス図・状態遷移図・タイミング仕様                |
| 8      | リソース消費目標     | CPU・メモリ・通信帯域・ストレージの割り当て目標        |
| 9      | 要件割り当て         | SWR → コンポーネントのマッピングテーブル               |
| 10     | トレーサビリティ     | SWR → ARC のマトリクス                                |
| 11     | 設計上の決定・根拠   | ADR（Architecture Decision Records）                  |
| 付録   | 変更履歴             | バージョン管理テーブル                                 |

#### 4.1.2 コンポーネント記述ルール

各コンポーネントは以下のフィールドを持つ。

| フィールド       | ルール                                                      |
|------------------|-------------------------------------------------------------|
| ID               | `ARC-<3桁連番>` 形式（例: ARC-001）                        |
| コンポーネント名 | 役割が明確にわかる名称（英字推奨）                          |
| 責務             | このコンポーネントが担う機能を2〜5行で記述                  |
| 担当 SWR         | 割り当てられた SWR-xxx をカンマ区切りで列挙                 |
| 外部依存         | 依存する外部ライブラリ・サービス・HW を列挙                 |
| 制約             | 実装制約・使用禁止 API 等                                   |
| インターフェース | 提供 API / 消費 API を区別して列挙                          |

#### 4.1.3 インターフェース記述ルール

| フィールド       | ルール                                                      |
|------------------|-------------------------------------------------------------|
| インターフェースID | `IFC-<3桁連番>` 形式                                     |
| 提供コンポーネント | ARC-xxx                                                    |
| 消費コンポーネント | ARC-xxx（複数可）                                          |
| 操作/メッセージ名 | 動詞＋目的語形式（例: `getTemperature()`）                |
| 引数・戻り値     | 型・範囲・単位を明記                                        |
| 呼び出し頻度     | 定期 / イベント駆動・周期（ms）を記載                       |
| エラー処理       | エラー時のふるまいを記述                                    |

#### 4.1.4 動的ふるまいの記述ルール

- シーケンス図: 正常シナリオ・異常シナリオの両方を必ず記述する
- 状態遷移図: すべての状態と遷移条件（ガード条件・アクション）を明示する
- タイミング制約がある場合は ms 単位で明記する

#### 4.1.5 Architecture Decision Record (ADR) の形式

```markdown
### ADR-<連番>: <決定事項タイトル>

**ステータス**: Accepted / Deprecated / Superseded

**コンテキスト**: <この決定が必要になった背景・課題>

**決定内容**: <採用したアプローチ>

**採用理由**: <選択した根拠>

**却下した代替案**:
- 案A: <説明> — 却下理由: <理由>

**影響**: <この決定がもたらすトレードオフ・制約>
```

## 5. トレーサビリティ要件

### 上位トレーサビリティ (SWR → ARC)

- すべての SWR は少なくとも1つの ARC コンポーネントに割り当てられること
- コンポーネントは自身が実現する SWR を必ず列挙すること

### 下位トレーサビリティ (ARC → SWE.3)

- SWE.3 完了後、各 ARC に対応する設計ユニット (UNT-xxx) を SAD に追記する

### トレーサビリティマトリクス形式

| SWR-ID  | SWR タイトル | ARC-ID  | コンポーネント名 | 備考         |
|---------|-------------|---------|-----------------|--------------|
| SWR-001 | ○○処理機能  | ARC-001 | ProcessManager  | 完全割り当て |

## 6. レビュー・承認基準

### レビューチェックリスト

- [ ] すべての SWR が少なくとも1つの ARC に割り当てられている
- [ ] すべての ARC に一意の ID が付与されている
- [ ] すべての ARC の責務が明確に記述されている
- [ ] コンポーネント間のインターフェースが網羅されている
- [ ] 正常・異常の両シナリオのシーケンス図がある
- [ ] 状態遷移図にすべての状態と遷移が記述されている
- [ ] リソース消費目標が定量的に記述されている
- [ ] 設計上の主要決定が ADR として記録されている
- [ ] トレーサビリティマトリクスに欠落がない

### 承認条件

1. 上記チェックリストの全項目が ✓ であること
2. 重大レビュー指摘がゼロであること
3. 工程責任者の署名（または承認記録）があること
