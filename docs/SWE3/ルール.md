# SWE.3 ソフトウェア詳細設計・ユニット構築 — ドキュメントルール

## 1. プロセス概要

### 目的

SAD で定義されたコンポーネントを**ユニット（最小実装単位）に詳細化**し、  
実装可能な詳細設計を確立する。その設計に基づきソースコードを実装する。

### 位置づけ（入出力）

```
入力:
  - ソフトウェアアーキテクチャ設計書 (SAD)
  - インターフェース設計書

出力:
  - ソフトウェア詳細設計書 (SDD)
  - ソースコード（ユニット）
  - コーディング規約

後続:
  - SWE.4（SDD・ソースコードを入力としてユニット検証へ）
```

#### 4.1.2 粒度ポリシーの宣言

**SDD 作成を開始する前に、本節のポリシーをプロジェクトまたは SDD 単位で決定し、ドキュメントヘッダーに記載する。**  
粒度が決まることで、設計者・レビュー者・実装者の期待値が揃い、手戻りを防げる。

##### クラス図・データ構造図の粒度レベル

| レベル | 名称         | 記載内容                                                                 | 典型的な使用場面                  |
|--------|--------------|--------------------------------------------------------------------------|-----------------------------------|
| **L1** | 概念レベル   | クラス名・関係のみ（属性・メソッドなし）                                 | 設計初期のスケッチ・SAD レビュー前 |
| **L2** | 仕様レベル   | L1 ＋ public 属性・public メソッド（引数名・型・戻り値型を含む）        | ASPICE SWE.3 の最低要件           |
| **L3** | 実装レベル   | L2 ＋ protected/private の全メンバー・コンストラクタ（そのまま実装可能） | SWE.4 前の完成状態                |

> ASPICE SWE.3 の成果物としては **L2 以上** を必須とする。  
> L3 を選ぶ場合、コード生成ツール（PlantUML → スケルトン自動生成）との連携を推奨する。

##### シーケンス図の粒度レベル

| レベル | 名称         | 記載内容                                                                         | 典型的な使用場面                  |
|--------|--------------|----------------------------------------------------------------------------------|-----------------------------------|
| **L1** | 概念レベル   | 参加者名・メッセージ名のみ（引数・戻り値なし）                                  | 設計初期・SAD レビュー前          |
| **L2** | 仕様レベル   | L1 ＋ 引数名・戻り値（型なし可）・activate/deactivate・alt/opt/loop ブロック    | ASPICE SWE.3 の最低要件           |
| **L3** | 実装レベル   | L2 ＋ 完全な型付きシグネチャ・private メソッド呼び出し・例外・タイミング制約   | SWE.4 前の完成状態                |

> **L3 で記載すべき private メソッド呼び出しの基準**: 複雑なロジックを内包する場合のみ記載する。  
> 単純な getter/setter レベルは省略してよい。

##### ポリシー宣言の記載箇所

SDD ヘッダー直下に以下の表を追加する。

```
| 図の種類     | 採用粒度 | 理由・備考                        |
|------------|---------|-----------------------------------|
| クラス図     | L2      | public API の確定を目的とするため  |
| シーケンス図 | L2      | 実装者への指示は API 仕様書で補う  |
```

#### 4.1.3 図の記法・必須/推奨区分

すべての図は **PlantUML 記法** (```` ```plantuml ```` ブロック) で記述する。  
PlantUML はテキストベースのため Git 管理・差分確認が可能であり、VS Code 拡張や CI での自動レンダリングに対応する。

| 図の種類             | 必須 / 推奨 | 適用条件                                    | PlantUML ダイアグラム種別  |
|----------------------|-------------|---------------------------------------------|---------------------------|
| モジュール構造図     | **必須**    | 常に（ユニット間依存関係の全体把握）         | `component` / `package`   |
| クラス図             | **必須**    | OOP 言語 (C++/Java/Python 等) を使う場合   | `class`                   |
| データ構造図         | **必須**    | C 言語等で struct/typedef を使う場合        | `class`（ステレオタイプ）  |
| シーケンス図（正常系）| **必須**   | ユニット間メッセージが発生するすべての機能   | `sequence`                |
| シーケンス図（異常系）| **必須**   | エラー・タイムアウト等の異常フロー           | `sequence`                |
| 状態遷移図           | **必須**    | 状態を持つユニットが 1 つでも存在する場合    | `state`                   |
| フローチャート       | **必須**    | 循環的複雑度 > 5 の関数・処理ロジック       | `activity`                |
| アクティビティ図     | 推奨        | 並行処理・フォーク/ジョインがある処理        | `activity`                |
| ER 図                | 推奨        | DB・永続化データ構造がある場合               | `entity`                  |

> **「必須」の判断基準**: 適用条件に合致するユニットが SDD 内に 1 つでも存在すれば  
> その図を SDD に含めなければならない。該当ユニットが存在しない場合は「対象なし — 理由: 〇〇」と明記する。

#### 4.1.4 モジュール構造図の記述ルール

- SDD 内のすべての UNT を矩形で示し、依存方向を矢印（`-->`）で示す
- 依存の種類（呼び出し / データ共有 / イベント通知）をラベルで区別する
- 外部ライブラリ・OS サービスはデータベース記号 (`database`) またはクラウド記号で区別する
- 粒度ポリシーの影響を受けない（常に UNT 単位で描く）

```plantuml
@startuml
package "対象コンポーネント (ARC-001)" {
    component "UNT-001\nRequestHandler" as UNT001
    component "UNT-002\nValidator"      as UNT002
    component "UNT-003\nDataStore"      as UNT003
}
database "FileSystem API\n(外部)" as EXT001

UNT001 --> UNT002 : 呼び出し
UNT001 --> UNT003 : 呼び出し
UNT002 ..> UNT003 : データ共有
UNT003 --> EXT001 : 呼び出し
@enduml
```

#### 4.1.5 クラス図 / データ構造図の記述ルール

関係の記号は以下を使い分ける。

| PlantUML 記号 | 意味             | 使いどころ                              |
|---------------|------------------|-----------------------------------------|
| `<\|--`       | 継承             | is-a 関係                               |
| `*--`         | コンポジション   | ライフサイクルを共にする強い包含関係    |
| `o--`         | 集約             | ライフサイクルが独立した包含関係        |
| `-->`         | 依存             | 一時的な使用関係（引数・戻り値で使用）  |
| `..|>`        | 実現             | インターフェース実装                    |

**OOP 言語（クラス図）— L2 の例（public メンバーのみ）:**

```plantuml
@startuml
class RequestHandler {
    + handleRequest(req : Request) : Response
    + cancel() : void
}
class Validator {
    + validate(req : Request) : ValidationResult
}
class DataStore {
    + save(record : Record) : bool
    + findById(id : String) : Record
}
enum RequestStatus {
    PENDING
    PROCESSING
    DONE
    ERROR
}
RequestHandler *-- RequestStatus
RequestHandler --> Validator  : uses
RequestHandler --> DataStore  : uses
@enduml
```

**OOP 言語（クラス図）— L3 の例（全メンバー、そのまま実装可能）:**

```plantuml
@startuml
class RequestHandler {
    - requestId : String
    - status    : RequestStatus
    - validator : Validator
    - dataStore : DataStore
    --
    + RequestHandler(v : Validator, ds : DataStore)
    + handleRequest(req : Request) : Response
    + cancel() : void
    --
    - validateInput(req : Request) : boolean
    - buildResponse(ok : boolean) : Response
}
class Validator {
    - maxLength : int
    --
    + validate(req : Request) : ValidationResult
    --
    - checkRange(value : int) : bool
    - checkFormat(str : String) : bool
}
class DataStore {
    - records  : List<Record>
    - filePath : String
    --
    + save(record : Record) : bool
    + findById(id : String) : Record
    --
    - serialize(record : Record) : String
    - persist() : bool
}
enum RequestStatus {
    PENDING
    PROCESSING
    DONE
    ERROR
}
RequestHandler *-- RequestStatus
RequestHandler --> Validator : uses
RequestHandler --> DataStore : uses
@enduml
```

**C 言語（データ構造図）— L2 の例:**

```plantuml
@startuml
class RequestData_t << (S,#ADD8E6) struct >> {
    + requestId : uint32_t
    + priority  : uint8_t
    + status    : RequestStatus_e
    + payload   : uint8_t[64]
}
class RequestStatus_e << (E,#FFD700) enum >> {
    REQ_PENDING
    REQ_PROCESSING
    REQ_DONE
    REQ_ERROR
}
class DataStore_t << (S,#ADD8E6) struct >> {
    + records : RequestData_t[MAX_RECORDS]
    + count   : uint8_t
}
RequestData_t --> RequestStatus_e : uses
DataStore_t   *-- RequestData_t   : contains
@enduml
```

#### 4.1.6 シーケンス図の記述ルール

- 正常系と異常系を**別々のシーケンス図**として作成する
- 参加者は UNT-ID を含む名称で宣言する
- `activate` / `deactivate` で実行期間を示す
- `alt` / `opt` / `loop` ブロックで条件分岐・繰り返しを表現する
- タイミング制約がある場合はメッセージラベルに `[≤ XXX ms]` を付記する

**シーケンス図（正常系）— L2 の例:**

```plantuml
@startuml
participant Client
participant "UNT-001\nRequestHandler" as UNT001
participant "UNT-002\nValidator"      as UNT002
participant "UNT-003\nDataStore"      as UNT003

Client -> UNT001 : handleRequest(req)
activate UNT001

UNT001 -> UNT002 : validate(req)
activate UNT002
UNT002 --> UNT001 : ValidationResult
deactivate UNT002

UNT001 -> UNT003 : save(record)
activate UNT003
UNT003 --> UNT001 : bool
deactivate UNT003

UNT001 --> Client : Response
deactivate UNT001
@enduml
```

**シーケンス図（正常系）— L3 の例（型付き・private 呼び出し含む）:**

```plantuml
@startuml
participant Client
participant "UNT-001\nRequestHandler" as UNT001
participant "UNT-002\nValidator"      as UNT002
participant "UNT-003\nDataStore"      as UNT003

Client -> UNT001 : handleRequest(req : Request) : Response
activate UNT001

note right of UNT001 : private メソッド呼び出し
UNT001 -> UNT001 : validateInput(req : Request) : boolean

UNT001 -> UNT002 : validate(req : Request) : ValidationResult
activate UNT002
UNT002 -> UNT002 : checkRange(value : int) : bool
UNT002 -> UNT002 : checkFormat(str : String) : bool
UNT002 --> UNT001 : ValidationResult(ok=true)
deactivate UNT002

UNT001 -> UNT003 : save(record : Record) : bool
activate UNT003
UNT003 -> UNT003 : serialize(record : Record) : String
UNT003 -> UNT003 : persist() : bool
UNT003 --> UNT001 : true
deactivate UNT003

UNT001 -> UNT001 : buildResponse(ok : boolean) : Response
UNT001 --> Client : Response(status=SUCCESS, code=0)
deactivate UNT001
@enduml
```

**シーケンス図（異常系）— L2 の例:**

```plantuml
@startuml
participant Client
participant "UNT-001\nRequestHandler" as UNT001
participant "UNT-002\nValidator"      as UNT002

Client -> UNT001 : handleRequest(req)
activate UNT001

UNT001 -> UNT002 : validate(req)
activate UNT002
UNT002 --> UNT001 : ValidationResult(NG, ERR_RANGE)
deactivate UNT002

UNT001 --> Client : Response(ERROR, ERR_RANGE)
deactivate UNT001
@enduml
```

#### 4.1.7 状態遷移図の記述ルール

- すべての状態を網羅する（到達不能な状態は `note` で明記する）
- 遷移ラベルは `イベント() [ガード条件] / アクション()` 形式で記述する
- 初期状態 (`[*]`) と終了状態を必ず明示する
- 状態が 7 つ以上の場合は複合状態（`state XX { }` によるサブ状態機械）への分割を検討する
- 粒度ポリシーの影響を受けない（常に完全な状態機械を描く）

```plantuml
@startuml
[*] --> Idle

Idle      --> Processing : start() [isReady] / initContext()
Idle      --> Error      : start() [!isReady] / logError()

Processing --> Done       : complete() / cleanup()
Processing --> Error      : timeout() / logTimeout()
Processing --> Processing : retry() [retryCount < MAX] / incrementRetry()

Done  --> Idle : reset()
Error --> Idle : reset() / clearError()

Done --> [*]
@enduml
```

#### 4.1.8 フローチャート（アクティビティ図）の記述ルール

- 関数単位で作成する（1 関数 = 1 フローチャート）
- PlantUML アクティビティ図記法（`start` / `stop` / `if` / `while`）を使用する
- すべての分岐に条件ラベルを付ける
- ループの終了条件を必ず示す
- 循環的複雑度 ≤ 5 の単純な関数は省略できる（「省略理由: 単純な直線処理」と記載）

```plantuml
@startuml
start
:受け取り: req;
if (req != null?) then (No)
    :errorCode = ERR_NULL_PTR;
    :返す: ERROR;
    stop
endif
if (validate(req) == OK?) then (No)
    :errorCode = ERR_INVALID;
    :返す: ERROR;
    stop
endif
:DataStore.save(record);
if (save 成功?) then (No)
    :errorCode = ERR_STORE;
    :返す: ERROR;
    stop
endif
:返す: SUCCESS;
stop
@enduml
```

#### 4.1.9 ユニット記述ルール

各ユニットは以下のフィールドを持つ。

| フィールド       | ルール                                                       |
|------------------|--------------------------------------------------------------|
| ID               | `UNT-<3桁連番>` 形式（例: UNT-001）                         |
| ユニット名       | コンポーネント名＋役割を示す名称                             |
| 親コンポーネント | 対応する ARC-xxx を記載                                      |
| 責務             | このユニットが担う処理を3行以内で記述                        |
| ファイルパス     | ソースコードの相対パスを記載                                 |
| 担当 SWR         | 実現する SWR-xxx をカンマ区切りで列挙                        |
| 公開 API         | 外部に公開する関数・メソッドの名称一覧                       |
| データ構造       | ユニットが使用する主要な構造体・クラスの概要                 |
| アルゴリズム     | 複雑な処理は擬似コードまたはフローチャートで記述             |

#### 4.1.10 API 仕様の記述ルール

```markdown
#### <関数名>

| フィールド | 内容                                 |
|-----------|--------------------------------------|
| 宣言      | `<戻り値型> <関数名>(<引数リスト>)`  |
| 目的      | <何をする関数か1行で>               |
| 引数      | <引数名>: <型>, <範囲/制約>, <説明> |
| 戻り値    | <型>, <値の意味>, <エラー時の値>    |
| 前提条件  | <この関数を呼ぶ前に満たすべき条件>  |
| 副作用    | <グローバル変数・状態への影響>      |
| エラー    | <エラーケースと戻り値・例外>        |
```

## 5. トレーサビリティ要件

### 上位トレーサビリティ (ARC → UNT)

- すべての UNT は対応する ARC を持つこと
- ARC は自身が実現する UNT 一覧を SDD に記録すること

### 下位トレーサビリティ (UNT → SWE.4)

- SWE.4 完了後、各 UNT に対応するテストケース (TC-UNT-xxx) を SDD に追記する

### コードのトレーサビリティ

- ソースコードの各関数/モジュールに SWR-ID または UNT-ID をコメントで記載することを推奨する

### トレーサビリティマトリクス形式

| ARC-ID  | コンポーネント名 | UNT-ID  | ユニット名         | ファイルパス |
|---------|----------------|---------|--------------------|-------------|
| ARC-001 | ProcessManager | UNT-001 | ProcessController  | src/proc.c  |

## 6. レビュー・承認基準

### レビューチェックリスト

**構造設計**
- [ ] すべての ARC が少なくとも1つの UNT に分解されている
- [ ] すべての UNT に一意の ID が付与されている
- [ ] モジュール構造図が存在し、すべての UNT と外部依存が記載されている
- [ ] クラス図（OOP）またはデータ構造図（C）に全クラス/struct が記載されている
- [ ] クラス図の関係（継承・コンポジション・依存）が適切に区別されている

**インターフェース・API**
- [ ] すべての公開 API に引数・戻り値・エラーが記述されている

**動的ふるまい**
- [ ] ユニット間相互作用に対してシーケンス図（正常系）が存在する
- [ ] ユニット間相互作用に対してシーケンス図（異常系）が存在する
- [ ] 状態を持つユニットに状態遷移図が存在し、すべての状態・遷移が網羅されている
- [ ] 循環的複雑度 > 5 の関数にフローチャートが存在する
- [ ] すべての分岐・ループに条件が明示されている

**エラー処理**
- [ ] エラー処理の設計が全ユニットに記述されている
